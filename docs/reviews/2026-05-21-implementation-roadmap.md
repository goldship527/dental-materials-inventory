### 優先4. ログインのレート制限と監査ログ

工数目安: M(1日)

なぜこの順番か:
- 医療系の販売では「ログインのブルートフォース対策」が必ず指摘される
- 監査ログがないと法人契約の情報セキュリティチェックリストに通らない
- 監査ログテーブルさえあれば、後続の機能で「誰がいつ何を変えたか」を追記しやすい

実装範囲:
- 失敗回数カウント(IP+メール単位、Postgresの簡易テーブル `login_attempts`)
- 5回連続失敗で15分ロック、ロック中はエラー画面で再試行時刻を表示
- 監査ログテーブル `audit_logs`(actor_user_id / action / target_type / target_id / details_json / created_at)
- Server Action の重要操作だけ監査ログを残す:
  - ユーザー追加・無効化・パスワードリセット(優先2の実装と合流)
  - 商品マスタ編集・新規作成
  - 発注先マスタ編集
  - 棚卸セッション破棄
  - 取り消し(REVERT)操作
- 管理者向け `/admin/audit-logs` 画面(直近100件閲覧、検索可)

Codex検証範囲:
- わざと5回失敗ログイン → ロックされる
- 15分後相当の時刻調整でロック解除を確認(ローカルではDB直接UPDATEで時刻を巻き戻して検証)
- 商品マスタ編集 → audit_logs に1件追加
- 管理者で /admin/audit-logs にアクセスして閲覧可能
- typecheck / build

ユーザー側後段作業: なし

- 在庫一覧のステータス列に新4段階を反映

Codex検証範囲:

Codex検証範囲:

## 進め方のルール

1. **1項目=1セッション=1コミット**を厳守。複数項目を1コミットに混ぜない
2. 各セッションは「仕様追記(spec.md) → 実装 → ローカル検証 → dev-log追記 → コミット&push」のワンセット
3. Codex は秘密値を要求しない。秘密値が要る検証はユーザー側手順として `docs/demo-deploy.md` に残す
4. 各セッションの最後にローカルで `pnpm typecheck` と `pnpm build` を通す
5. 認証や決済のように副作用のある処理は、push前にローカル `pnpm dev` で通し動作確認を必須にする
6. 完了したら本ファイルの該当節に「完了: 2026-MM-DD」と追記する
