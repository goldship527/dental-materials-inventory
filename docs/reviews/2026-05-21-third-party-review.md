# 一般歯科材料在庫管理システム MVP 第三者レビュー

レビュー対象: `dental-materials-inventory`(2026-05-20時点 / フェーズ3-Rまで実装済み)
レビュー方法: 仕様書(spec.md)・README・project-manual.md を読み込み、業務アプリ観点で評価
レビュー範囲: コードの細部レビューは行わず、業務適合性・販売前の不足・運用リスクに重点

---

## 1. 全体評価

結論を先に書きます。**「個人開発のMVPとしては明らかに完成度が高く、画面構成・履歴設計・安全設計の方針がよく練られている」一方で、「歯科医院に有料で売る業務アプリ」としては、まだ販売前に直すべき致命的ポイントが2〜3個と、磨き込みポイントが多めに残っている**、というのが正直な評価です。

良いところ:

- 仕様書がフェーズごとに整理されており、「今は作らない」を明示している点は、業務アプリとして非常に健全な姿勢です。歯科医院のような医療系小規模事業所向けでは、機能を絞って間違いを起こさない作りのほうが信頼されます。
- 「バーコードを読み取っただけでは在庫を変更しない」「確定ボタンを押さないと在庫は動かない」という安全設計は、現場の運用感覚に合っています。誤スキャン地獄を起こさない設計はとても大事です。
- `StockMovement` を全在庫変更の単一ハブとし、操作元(`sourceType`)、取消(`REVERT`)、棚卸セッション由来(`STOCKTAKE_SESSION`)を区別している点は、後の監査要件・保守に強い基盤です。
- マルチテナント(Organization / Clinic)、複数バーコード(ProductBarcode)、棚卸セッションの途中保存・破棄・確定までを最初から設計に入れているのは、後から逆ハードコードするより圧倒的に楽です。
- 「実在製品データ、患者情報、実在クリニック名をseed/Gitに入れない」という運用ルールが仕様書の中に明文化されているのは、医療系の販売では強力な「事故が起きにくい設計の根拠」になります。

気になるところ(後段で詳述):

- 商品写真がローカルファイル保存のままで、Vercel本番デプロイでは事実上消える(販売前に必ず直すべき)。
- ロット番号・有効期限の保存が無く、表示のみに留まっている。歯科材料(特に薬剤・印象材・接着剤)は期限切れ厳禁なので、販売訴求点として弱い。
- 商品マスタの一括取り込み(CSV)がないため、初期導入時に200〜500品を手入力する負荷が地獄になりやすい。
- 発注先の連絡先(電話・メール・住所・担当者)が編集できないため、発注フロー(印刷→FAX or 電話)が画面の外で止まってしまっている。
- ユーザー追加・退職者アカウント停止・パスワードリセットがなく、共用アカウント前提のため、本番運用ルールが組めない。

全体としては、**「販売可能なベータ」になる手前ですが、3つ4つの致命点を潰せば1医院向けに月額1〜2万円程度で売り始める評価には到達できる**、という位置づけだと思います。

## 2. 業務フロー上の不足点

業務フローを「在庫見える化 → 補充判断 → 発注 → 納品 → 入庫 → 出庫 → 棚卸 → 履歴確認」と並べると、現状のMVPは「補充判断」までと「出庫・棚卸」のところはしっかり作られているのに対し、**「発注の出口」と「納品の入口」が画面の中で繋がっていない**のが一番大きな不足点です。

具体的に挙げます。

1. 発注書/連絡手段が存在しない。「発注候補一覧をA4印刷」までは行けますが、その先の「発注先にどう伝えるか」が画面の中にありません。歯科医院の現実では、発注先(歯科商社/メーカー)へはFAX・電話・メール・営業担当アプリ(GC、ヨシダ等の専用)が混在します。最低限「メールで送る」「PDFで保存して添付する」のどちらかは欲しいです。
2. 納品確認のフローが無い。発注候補に対して「実際に届いた」「届いてないので催促する」「数量が違った」を扱えるテーブルがないため、発注候補がいつまでも `DRAFT` か `CONFIRMED` のまま残ります。仕様書でも「発注書単位の親子構造」「納品確認」が未実装と明記されていますが、これがないと業務ループが閉じません。
3. 包装単位の換算が無い。歯科では「箱(20本入)で発注、現場では1本ずつ消費」が一般的です。「現在庫:3」が「3本」なのか「3箱」なのかが曖昧で、バーコード出入庫で「箱を読んで、中の本数で減らす」という操作が今のままだとできません。これは将来必須です。
4. ロット番号・有効期限がDB保存されない。GS1から解析して画面表示はするが、`BarcodeScanLog` には保存(フェーズ3-N)で読取履歴に残るのみ。`StockItem` 側でロット別在庫を持たないため、「期限が近い在庫を先に使う」「期限切れ在庫を発見する」ができません。歯科材料は薬機法上の管理対象も多く、ここが弱いと医院長に「うーん」と言われやすいです。
5. 退職者アカウント停止が無い。歯科助手の出入りはどの医院でも頻繁です。`User` の論理削除(`isActive`)とログイン拒否は必須レベルです。
6. 複数クリニック切替UIが無い。法人格で2医院・3医院持っている院長は多く、画面の上でクリニックを切り替えられないと「事務さんは1日の中で何度もログインし直す」運用になります。
7. 「在庫がない商品を新規追加して即在庫を1にする」ボタンが無い。商品マスタ新規作成では `StockItem` を作らない設計のため、「新しく届いた材料を登録して数量を入れる」までに2画面跨ぐ必要があります。現場のスピード感に合っていません。
8. 発注先マスタの新規作成・連絡先項目編集が無い。発注先名のみ編集可能で、住所・電話・メール・担当者が登録できないため、「発注先詳細」画面の情報量が薄く、ハブとしての価値が下がっています。
9. 棚卸の差異理由が個別に入らない可能性がある。`StocktakeSessionItem` にメモはあるが、「差異あり=破損、紛失、伝票忘れ」のカテゴリ別集計が無いと、経営者目線で「ロスの原因」が分かりません。

## 3. UI/操作性で気になる点

仕様書をベースに、デモ画面が想定通りに作られていると仮定して、現場目線で気になる点を挙げます。実機(タブレット/スマホ)とデモログイン情報を共有いただければ、より具体的な指摘ができます。

- ナビゲーション項目が多い(ホーム、在庫一覧、商品、発注先、バーコード、取込確認、カード、不足一覧、発注候補、履歴、マニュアル、アカウント、棚卸)。仕様書には「画面幅が狭い場合は横スクロール」と書かれていますが、現場のタブレットでは横スクロール式ナビは押し間違えやすく、特に手袋越し操作だと困難です。「日常モード(在庫/カード/バーコード/棚卸)」と「管理モード(商品/発注先/履歴/取込/アカウント)」の2階層化を検討する価値があります。
- 「よく使う商品カードの `+1 / -1`」と「バーコード出入庫の確定ボタン」が併存するため、スタッフが「結局どこで在庫を減らせばいいのか」迷いやすい予感がします。スタッフ向けマニュアルで明確に「箱の出し入れはバーコード、目視確認の補充はカード、修正は在庫編集」など役割分担を切り分けて教える必要があります。
- 不足在庫一覧から発注候補への追加 → 発注候補で数量変更 → 印刷、というフローは良いのですが、**「発注候補が確定済みになったあと、誰がいつ実際に発注したかが分からない」**点がUI上で見えにくいです。`CONFIRMED` の意味が「発注済」なのか「発注しようと決めた」なのかが曖昧です。
- バーコード出入庫の理由ボタン(使用 / 棚卸調整 / その他、納品 / 返品戻り / 棚卸調整 / その他)はシンプルで良いです。ただ「棚卸調整」をバーコード出入庫から行えると、棚卸セッションと履歴の整合が取りづらくなる懸念があります。「棚卸調整」はセッション経由に限定し、バーコード出入庫からは外したほうが事故が減ります。
- 数量編集時の理由メモが必須になっている点は良いです。ただし「理由メモ自由入力」だと、慣れたスタッフほど「修正」「補正」と書きがちで、後から監査するときに役に立ちません。理由ボタン(誤入力修正 / 破損 / 紛失 / 期限切れ廃棄 / その他) + 補足メモ任意 の構造に揃えるべきです。
- 取り消し(REVERT)ボタンが履歴一覧の各行に常に見えると、誤押しが怖いです。「直近の自分の操作のみ取消可能」「2クリック確認モーダル」「取消の理由メモ必須」あたりまでガードしたほうが安全です。仕様書に「条件を満たす場合だけ」と書かれていますが、表示位置・色・確認モーダルの文言を慎重に作り込む必要があります。
- ホームのスパークライン「14日推移は現在不足件数を複製した簡易表示」は、利用者から見ると「日次の変動」と誤解する可能性が高いです。注釈は仕様書に書かれていますが、画面では「今後対応予定」と書くより、当面は表示しないほうがクレームを生みにくいです。
- 商品写真が「写真なしの控えめな枠」と仕様にありますが、写真未登録の商品が大半の初期状態では、画面全体が「写真なし」だらけになり、よく使う商品カードが視覚的に賑やかになりません。**初期段階では写真なしで成立する設計**にしておくか、カテゴリ別のデフォルト絵文字/アイコンで埋めるなどの工夫が要ります。

## 4. 販売前に必須だと思う機能

「これがないと売れない」ものを優先度高い順に挙げます。

1. 商品写真の永続化(オブジェクトストレージ対応)
   現状はローカルファイル保存で、Vercel本番では再デプロイやコールドスタートでファイルが消えます。Supabase Storageへの移行が最も近道です。**販売直前にこれが残っていると致命傷**なので、最優先で潰してください。
2. 商品マスタCSVインポート(または対話的バルク登録)
   1医院あたり通常200〜800品ほどの材料を扱います。これを画面から1件ずつ手動登録は現実的ではありません。CSVが面倒なら、Excelのコピペを受け付ける貼り付け一括登録でもよいです。CSV取り込みがないと、有料導入の初日にスタッフが半日潰れる事態になります。
3. 発注先の連絡先・担当者管理 + 発注書PDFまたはメール送信
   「発注候補から印刷だけ」では業務が閉じません。最低限PDFダウンロード(発注先別)、欲を言えばメール送信(発注先メールアドレス宛、コピーは医院側にも)があれば、業務ループが完結します。
4. ユーザー追加・削除(無効化)・パスワードリセット
   退職者アカウントを止められない医院ITは無いです。本人パスワード変更だけでは管理者運用ができません。
5. ロット番号・有効期限の在庫保存と「期限が近い」一覧
   GS1解析と読取履歴保存は実装済みなので、`StockItem` をロット別に持つか、`stock_lots` 相当のテーブルを追加して、期限管理を1段だけでも入れたほうが、医院向けに大きな差別化になります。
6. 監査ログ(誰がいつ何を変えたか)の最低限保存
   `StockMovement` は在庫変更しか追わないため、「商品マスタを編集した」「発注先を編集した」「ユーザーを停止した」などの監査が抜けます。`audit_logs` テーブルを1つ作り、Server Action の重要操作だけログを残す形で十分です。
7. バックアップと復旧手順の文書化(契約書付録レベル)
   Supabase の `pg_dump` 定期取得、復旧テスト、保管場所、保管期間を運用ルールとして明文化し、お客さんに見せられる体裁にしてください。

これらが揃えば、「**1医院単独・小規模オンプレ代替**」の用途で月1〜2万円帯で売れる土台になります。

## 5. 後回しでよい機能

販売前に必ず必要ではないもの。

- 発注書の親子構造(`purchase_orders`)。発注候補→発注書化のテーブル分離は、運用が回り始めてから着手で十分。最初は「発注候補が確定済みになったら、CSV/PDFを出して終わり」で乗り切れます。
- 外部システム連携(レセコン、会計、受発注EDI)。これは販売後の上位プランの目玉として温存。
- 複数クリニック切替UI。最初の顧客が単院ならまったく不要。「将来複数医院対応に拡張可能」とDB側に `clinicId` を持っているだけで十分な売り文句になります。
- PWA・オフライン対応。歯科医院の院内Wi-Fiは基本切れない前提でOK。タブレット最適化(レスポンシブ・タップ領域広め)のほうが先です。
- 多要素認証。クリニック単位のローカル運用ではまだ過剰。ただし、メール認証によるパスワードリセットの導線は必要です。
- AI予測発注、自動補充提案、在庫推移グラフの本格版。販売後の差別化機能として温存。
- バーコード読み取りによる「即時出庫」モード。現状の「確定ボタン必須」を維持したほうが事故が少ないので、無理に追加しないほうが良いです。
- 詳細な権限管理(役割×アクション マトリクス)。最初は「管理者 / スタッフ」の2ロールで十分です。

## 6. 技術・運用面のリスク

仕様書を読んだ範囲での技術・運用リスクを挙げます。

セキュリティ・データ保護:

- NextAuth.js + Credentials Provider の運用は、`AUTH_SECRET` が漏れた瞬間に全セッションを偽造できるため、定期ローテーションと環境変数管理が運用書に明記されている必要があります。仕様書には「Git/READMEに書かない」と書かれていますが、ローテーション手順が project-manual.md に欲しいです。
- パスワードハッシュは bcrypt salt rounds 12 で適切ですが、ログインのレート制限・失敗回数ロックが見当たりません。Supabase 公開エンドポイントなのでブルートフォース対策は必須です。最低限 NextAuth の `signIn` callback で IP+メール単位の失敗カウントを入れるべきです。
- 商品写真のローカル保存は前述のとおりVercel本番では永続しません。これは販売前必須項目。
- 在庫変更の楽観ロック(増減の原子性)は `increment` で対応とありますが、「直接編集」では `update` で値を上書きする可能性があります。同時編集すると後勝ちで在庫がずれます。`updatedAt` を使ったバージョン照合か、フォーム送信時に `expectedQuantity` を渡してサーバー側で再確認するほうが安全です(棚卸セッションの確定処理ではすでにこの方式が採用されているので、在庫の直接編集にも同じ思想を広げてください)。
- `StockMovement` の `REVERT` 後にさらに取り消すと「取消の取消」が連鎖する可能性があります。仕様上は `revertOfId` が空でないと取消対象外なので保護はされていますが、UIで「取消済み」「取消の履歴」の区別をはっきり出さないと、運用者が混乱します。

データ整合性・在庫ずれ:

- 棚卸セッション確定時、開始時 `expectedQuantity` と現在庫の差を検出して止める仕組みは良いです。ただし「棚卸途中で別スタッフがバーコード出入庫した場合」のメッセージを丁寧に出さないと、現場が止まります。
- バーコード出入庫の二重送信防止は仕様には明記されていますが、ネットワークタイムアウト後の再送(ユーザーが連打)に対するアイデムポテンシキーの仕組みがあるかを確認したいです。Server Action のリクエストID(クライアントUUID)で重複を弾くのが堅いです。

Vercel + Supabase 構成の懸念:

- Vercel のサーバーレス関数はファイル書き込みが揮発するため、商品写真ローカル保存は本番では成立しません(再掲)。Supabase Storage または Cloudflare R2 への移行が必須。
- Vercel Hobby プランのbuild時間制限・関数実行時間制限(10s)に、棚卸セッション確定や大量CSV取り込みで触れる可能性があります。Pro プラン前提で見積もるべきです。
- Supabase の無料/Pro 枠でのコネクションプール(PgBouncer)の使用が必須です。Prisma で `?pgbouncer=true&connection_limit=1` を `DATABASE_URL` に付けないと、Vercel の関数同時実行でコネクション枯渇しがちです。
- Supabase 無料枠は1週間アクセス無しで一時停止します。デモ環境を「営業中ずっと起動」したいなら Pro 必須。
- バックアップは Supabase Pro で日次自動取得されますが、長期保管(数ヶ月単位)は別途設計が必要です。
- 写真が Supabase Storage に移ったあとの「アクセストークン付き署名URL」配信パターンを早めに検証してください。Next.js から `<Image>` で直接表示する場合、毎回サインドURL生成のコストが地味に効きます。

権限・組織管理:

- いま `requireActiveClinic` でクリニックスコープを縛っているのは堅いですが、`organizationId` 横断のクエリ(管理者ユーザーが2クリニックを同時に見たい)に切り替える日が来ます。今のうちに「クリニックスコープ」と「組織スコープ」のヘルパーを分けて整理しておくと、後で楽です。
- 退職者対応で `User.isActive = false` 切替を入れ、NextAuth の `authorize` でフェイルさせる仕組みは1日仕事で入ります。販売前にぜひ。

## 7. 優先して直すべき順番

販売判断までの工数最短ルートとして、以下の順をおすすめします。

1. 商品写真のオブジェクトストレージ移行(Supabase Storage)
   理由: Vercel本番でいきなり死ぬため。販売直前に発覚すると差し戻し確定。
2. ユーザー追加・無効化・パスワードリセット(管理者画面)
   理由: 退職者対応がないと法人契約の話が前に進まない。
3. 商品マスタCSV/Excel貼り付け一括取り込み
   理由: 初期導入の体験を激変させる。導入0日目に売り手のサポート工数を減らす意味も大きい。
4. ログインのレート制限と失敗ロック + 監査ログ最低限
   理由: 医療系のセキュリティ要件で必ず指摘されるため。
5. 発注先の連絡先(電話/メール/住所/担当者)編集 + 発注書PDF出力
   理由: 発注の出口がないと業務ループが閉じない。
6. 在庫直接編集の楽観ロック(`expectedQuantity` 再確認 / `updatedAt` バージョン照合)
   理由: 棚卸では実装済みなのに直接編集にないと、ここから事故が起きる。
7. ロット番号・有効期限の在庫側保存と「期限間近一覧」
   理由: 歯科向け差別化要素。販売後でも良いが、デモで見せられると刺さる。
8. タブレット/スマホ向けレイアウト調整(横スクロールナビ廃止、タップ領域拡大、入力時のソフトキーボード対策)
   理由: 現場で実際に使われる端末で使えないと話にならない。
9. 「棚卸調整」をバーコード出入庫の理由から除外
   理由: 棚卸セッションと履歴の責任境界を明確化する小修正。事故防止。
10. 包装単位の換算(箱→本)
    理由: 重要だが、最初の顧客に「使い始めて1〜2ヶ月以内」と説明できる程度の後回しでも許容範囲。

順番の意図は、**「販売前に必須(1〜4)」「業務ループを閉じる(5〜6)」「差別化と現場フィット(7〜10)」**の3層構造です。

## 8. 想定価格・売り方についての所感

仕様書とproject-manualの料金イメージ(初期20〜30万円、月3〜5万円、5クリニック対応で月4.5万円)について、第三者目線での所感です。

価格レンジについて:

- インフラ原価が Vercel Pro + Supabase Pro で月7,000〜8,000円程度(契約条件で前後)。サポート・バックアップ確認・軽微修正の人件費を上乗せして月3〜5万円というのは、**歯科医院向け業務SaaSとしては中央値**だと思います。日本の歯科向け業務システム(レセコン、予約システム)は月3〜10万円帯が標準なので、価格帯としては妥当です。
- ただし、「在庫管理だけ」で月3万円を取るのは、競合(無料Excel運用、ニッシン/松風などの商社が提供する簡易在庫アプリ、汎用在庫アプリ ZAICO の月3,000〜10,000円)と比べると割高感があります。**「歯科特化」を明確に打ち出さないと、ZAICOで十分という結論にされやすい**です。
- 「歯科特化」の根拠としては、ロット/期限管理、JMDN対応、GS1解析、歯科商社マスタの取り込みプリセット、よく使う材料カテゴリ(印象材、根管材、接着材、補綴材)のプリセットなどが効きます。現状の機能だけだと「汎用在庫アプリに歯科の語彙を載せた」ように見えてしまうので、ここの説明資料が大事です。

売り方について:

- 最初の顧客はおそらく「ITに興味のある若手院長」が1医院。**月1万円 / 初期5〜10万円程度で、6ヶ月モニター契約**から始めるのが現実的です。事例化させて、その後正規価格に上げる動きが妥当だと思います。
- 5クリニック法人を最初の顧客にする場合、複数クリニック切替UIと監査ログが現状不足なので、「半年後にこの機能を入れる」というロードマップ提示が必須になります。
- 営業ルートとしては、(a) 歯科商社の営業担当との提携(商社が顧客を持っている)、(b) 歯科向けITコンサル/会計事務所との提携、(c) Twitter(X)等で若手院長コミュニティに発信、の3ルートが現実的です。直接訪問営業は時間効率が悪いです。
- 「無料デモ → 試用1ヶ月 → 正式契約」の体験設計が大事で、現状の公開デモが Vercel + Supabase で動いていることはこの戦略にとても向いています。デモにそのままサインアップして使い始められる導線(セルフサーブ無料トライアル)を作れると、営業効率が一気に上がります。
- ターゲットを最初から絞るなら、**インプラント・矯正・予防歯科などの自費比率が高い医院**が候補です。理由は、これらの医院は材料単価が高く、ロス・期限切れの金額インパクトが大きいため、在庫管理ツール導入のROIが計算しやすいからです。一般歯科よりも提案が刺さりやすいと思います。

価格・売り方に関するリスク:

- 「歯科特化」を看板にして売るには、薬機法上の管理(滅菌物・医療機器分類)の文言を明確に「対象外」と書かないと、医院長に「うちは滅菌物の管理もこれでやりたい」と言われて後でトラブルになります。販売資料で対象範囲・対象外をはっきり分ける必要があります。
- 「クラウド」と「データ保管」への抵抗は、年配の院長には依然強い領域です。「データは日本のSupabase Tokyo Region に保存」「バックアップは何日保持」「契約終了時のデータ返却方法」を販売資料で明文化したほうが受注率が上がります(逆にこれを書かないと法人取引で詰まります)。

## まとめ

開発者(個人開発)のMVPとしての完成度は高く、安全設計・仕様書の質・フェーズ管理の進め方は、有料販売を見据えるレベルに十分達しています。一方で、**「商品写真の永続化」「ユーザー追加・無効化」「CSV取り込み」「発注書PDF出力」の4点**は販売前に必ず潰すべきです。これらが揃えば、月1〜2万円帯の小規模1医院向けプランで売り始める判断が現実的に可能です。

5クリニック法人や月3〜5万円帯を狙うのであれば、「監査ログ」「複数クリニック切替UI」「ロット/期限管理」「タブレット最適化」を半年〜1年のロードマップで提示できる体制があれば話が前に進みます。

デモ画面とログイン情報をいただければ、画面実装レベルでの追加レビュー(タッチ操作・印刷レイアウト・色のコントラスト・空状態の見え方など)も可能です。
