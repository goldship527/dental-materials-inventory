## A. 致命的(販売前に必ず直す)

### A-1. バーコード出入庫 `/barcode/stock` が公開デモに存在しない(404)

確認した事実:

- `https://...vercel.app/barcode/stock` にアクセスすると「404 This page could not be found.」が表示される
- `/barcode` ページにも「出入庫へ」ボタンが存在しない
- 仕様書 §40「フェーズ3-R バーコード出入庫」とdev-log 2026-05-20「バーコード出入庫MVP実装」エントリーには、実装済みと記載がある

結論:

公開デモは**ローカル開発の最新コミットより数日遅れた状態でデプロイされている**可能性が高いです。仕様書の目玉機能が公開デモで見せられないのは、第三者レビューや営業デモの致命傷です。

確認 + 対応:

1. ローカルの最新コミットをGitHubにpush済みかを確認
2. Vercelダッシュボードで `dental-materials-inventory` プロジェクトの最新デプロイ日時を確認
3. 自動デプロイが有効になっているか確認(GitHub mainブランチへのpushでVercel自動再デプロイされる設定)
4. Supabase の `corepack pnpm db:push` が最新マイグレーション(StocktakeSession等を含む)で実行済みかも確認
5. 再デプロイ後、もう一度 `/barcode/stock` にアクセスして表示されることを確認

### A-2. ユーザー名の文字化けが全画面で発生

確認した事実:

文字化けの文字列 `"繝・Δ諡・ｽ・` が、以下の画面の「操作者」「ログイン中ユーザー」表示で**全箇所に出ている**:

- `/home`: 「`"繝・Δ諡・ｽ・` としてログイン中」
- `/movements`: 全7件の「操作者」列
- `/stocktake/sessions`: 進行中セッションの「開始者」列

原因の推定:

これはバイト列としては、おそらく `テストユーザー` または `自分` が **UTF-8で書かれた文字列をShift-JIS/CP932として再解釈した**結果に近い壊れ方です。dev-log 2026-05-20「seedユーザー表示名の調整」で「`自分` から `テストユーザー` に変更」とあり、後続のdev-log「ローカルログイン復旧」では「表示名を `テストユーザー`」と更新したログがあります。

最も疑わしい原因:

1. Supabase本番DBに対する `corepack pnpm db:seed` を実行した環境(PowerShell?)のコンソールエンコーディングと、Node.js が読む既定エンコーディングが食い違っている
2. Vercelの環境変数 `DEMO_USER_NAME` がVercelダッシュボードで全角文字を含んだまま設定されており、Vercelの環境変数保管時のエンコーディングが崩れた
3. seedスクリプトで `String.prototype.normalize()` や `Buffer.from(..., 'binary')` 経由で文字列を扱っている箇所がある

対応:

1. Supabase Studio にログインし、`users` テーブルの `name` カラムを直接見る
2. もし壊れた値が入っていれば、UPDATE で `テストユーザー` に直接書き換える
3. Vercel 環境変数 `DEMO_USER_NAME` の値を Vercel ダッシュボードで `テストユーザー` に再設定する
4. seed.tsで `process.env.DEMO_USER_NAME ?? 'テストユーザー'` をそのまま使うのではなく、`Buffer.byteLength(name, 'utf8')` でバイト数を検証するなど、エンコーディング不一致を検知するチェックを足す
5. seed実行は `corepack pnpm db:seed` を CMD ではなく PowerShell から実行する場合は `chcp 65001` を先に流す(README/project-manualへの追記)

### A-3. /login 画面に開発者向け文言が残存

`/login` 画面に以下の文言が表示されています:

## B. 業務UX上の重要指摘

### B-1. 不足在庫一覧で「不足数 0」の商品が表示される

確認した事実:

`/shortage` で「患者用エプロン 青」が表示されているが、現在庫6・最低在庫6・**不足数0** と表示される。

判定ロジックは仕様書通り `quantity <= minStock`(等号含む)です。仕様通りなのですが、現場スタッフ目線では「不足してない商品が不足一覧に出ている」と混乱します。

提案:

選択肢A: 等号を含まない `quantity < minStock` に変更し、ぎりぎりは不足扱いしない
選択肢B: 等号を含んだまま、ステータスを「不足」と「ぎりぎり」「在庫=最低在庫」に分けて表示
選択肢C: 不足数0の行はソートで一番下に固定し、視覚的に弱いトーンで表示

販売前に**選択肢Bが現場のリテラシー的におすすめ**です。

### B-2. 在庫一覧で全商品のフォームが常時展開

確認した事実:

`/inventory` を開くと、50商品それぞれに「新数量」「理由メモ」「更新」の編集フォームが常時展開されて並んでいます。1商品ごとに縦約100pxを占有するため、画面下に行くほどスクロール量が膨大です。

問題点:

- 誤って隣の商品のフォームに入力するリスクが高い
- スクロール中に「更新」ボタンを誤タップする可能性
- 500品クラスになると、レンダリングコストが無視できないレベルになる

提案:

- 各行に「編集」ボタンだけを表示し、押した行だけインライン展開する
- または、行クリックでモーダルを開き「現在庫」「新数量」「理由メモ」を入力する形にする
- 編集中に他の行はクリックで切り替わるまでフォームを出さない

### B-3. 「在庫0」と「不足」がステータスとして同じ表記

確認した事実:

`/inventory` でグローブM(在庫0)・グローブL(在庫0)・サージカルマスク(在庫0)・フェイスシールド(在庫0)も、グローブS(在庫1)・仮封材(在庫2)も、ステータスは一律「不足」になっています。

問題点:

- 在庫0は「いまから患者対応できない」の緊急事態
- 在庫1〜最低在庫の間は「補充判断したい」段階
- この2つを同じステータスにすると、不足一覧で本当に止まっているものが埋もれます

提案:

- ステータスを `在庫切れ`(0件) / `不足`(1〜最低在庫) / `十分`(最低在庫超) の3段階に分ける
- ホームの不足バッジでも、内数として「うち在庫0件」を併記(既に実装されていることを確認)
- 不足一覧でも「在庫切れ」を最上位にソート

### B-4. 棚卸セッションのデモデータが「進行中だけ」

確認した事実:

`/stocktake/sessions` に進行中セッション(05/20 11:43開始、50商品全未入力)が1件あるだけで、確定済み・破棄済みの履歴がゼロです。

問題点:

- 第三者がデモを見たとき、「棚卸の確定が動くアプリかどうか」が画面上から判断できない
- 仕様書 §32.7 で実装した「履歴詳細」が、デモ上では何も表示できない

提案:

- seedスクリプトに、確定済みセッション1〜2件、差異あり明細・差異なし明細・スキップ明細を含むデモデータを足す
- 進行中セッションも、未入力50件のままではなく「30件入力済み、5件スキップ、15件未入力」のように途中状態にしておくと、画面の機能が一目で伝わる

### B-5. 共通ナビ14項目を1段にまとめている

確認した事実:

ナビ項目: ホーム / 在庫一覧 / 商品 / 発注先 / バーコード / 取込確認 / カード / 不足一覧 / 発注候補 / 履歴 / マニュアル / アカウント / 棚卸 / ログアウト = **14項目を1段の横スクロールバーで表示**。

問題点:

- タブレット(1024px幅前後)では横スクロールが必要になり、隠れた項目を見つけにくい
- 「取込確認」「カード」のように利用頻度が日常では低い項目と、「在庫一覧」「不足一覧」のような毎日使う項目が同列に並んでいる
- 「マニュアル」だけ別系統(運用文書)なのに、ナビに混ざっている

提案:

- 日常モード(ホーム/在庫一覧/カード/バーコード/棚卸/不足一覧/発注候補)を上段固定
- 管理モード(商品/発注先/取込確認/履歴/マニュアル/アカウント)を「メニュー」ハンバーガーまたは2段目展開
- ログアウトはアカウントの中に格納

販売前のキラーUX改善になります。

## C. デモ運用上の懸念

### C-1. Vercel プレビューURLでの公開

確認した事実:

URL に `-lz5ieuydh-` というデプロイハッシュが含まれており、これはVercelの**プレビューデプロイ用URL**です。

問題点:

- 新しいコミットでデプロイするたびにURLが変わる(顧客のブックマークが切れる)
- プレビューデプロイは既定で Vercel SSO 保護がかかる場合がある(今回は外せていたので確認は不要)
- 「lz5ieuydh」のような乱数文字列が含まれるURLは、第三者に共有する際の心理的ハードルが高い

提案:

- Vercelダッシュボードで `dental-materials-inventory.vercel.app` のような**Production URL**を割り当てる
- もしくは独自ドメイン(例: `demo.dental-mvp.com`)を割り当てる
- 営業資料・名刺・契約書のいずれかにURLを書く場合は、必ず固定URLで

### C-2. デモDBに進行中の棚卸セッションが居座っている

確認した事実:

棚卸セッションが2026-05-20に開始されたまま、入力ゼロで「進行中」のままになっています。デモで「新規開始」ボタンを押すと、仕様書通り**新規作成されず既存セッションに誘導**されます。

問題点:

- 第三者が「棚卸を最初から見たい」と思っても、既存セッションを破棄しないと新規開始できない
- 破棄UIで破棄してしまうと、デモを次に見る人はやはり棚卸の初期状態を確認できない

提案:

- デモDBに対して定期的な seed リセットを Vercel Cron Job または Supabase 関数で動かす
- 「デモリセット」ボタンを `/account` 配下に1つだけ用意し、デモログイン中だけ表示する(本番ユーザーには表示しない)
- もしくは、デモ用クリニックを2つ用意し、片方は「棚卸進行中」、もう片方は「未開始(新規スタート可能)」にする

### C-3. 仕様書と実機の不足件数の不整合

確認した事実:

- 仕様書 §0.1 と project-manual の §12「公開デモDBの初期化状況」: 「不足商品10件」
- 実機の `/home` `/shortage`: **不足15件**

仕様書通りに進めていれば10件のはずなので、仕様書側が古いか、後でseedを変更したかのどちらかです。販売用資料として両者が一致していないと不信感を生むので、どちらかに揃えてください。

## D. 良かった点(残しておきたい)

実機で確認して、特に良かった点を列挙します。販売資料に書ける訴求点として残しておくと良いです。

- ログイン → ホームの遷移が早く、画面表示が軽い(Next.js App Routerのサーバー側レンダリングが効いている)
- `/shortage` から `/orders` への「候補へ追加」「追加済み」表示が、重複追加を防ぐUIとして正しく動く
- `/orders` のサプライヤー別グルーピングと、状態(未確認/確認済み/見送り)の3段階管理が業務感覚と合う
- `/movements` の「取り消し済み」表示が、過去履歴と現在の関係を明示できている
- `/movements` の操作元フィルタが「在庫一覧 / よく使うカード / 棚卸 / 棚卸セッション / 履歴取り消し」と分かれており、後で監査するときに使いやすい
- `/inventory` のステータスラベル(不足/十分)が視覚的に控えめで、表示が落ち着いて見える
- `/products/new` に「患者情報、実在クリニック名、会社の秘密情報、APIキーやパスワードは入力しないでください」の注意文がきちんと出ている
