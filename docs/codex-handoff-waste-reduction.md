## 3. 仕様上の制約（強化機能を考える前提）

以下は `docs/spec.md` で確定している方針です。新機能はこれを破らないこと。

| 制約 | 内容 | 関連箇所 |
|---|---|---|
| 在庫数は人が確定 | バーコード読み取りや購入履歴一致だけで在庫数を変更しない。確定ボタンが必要 | spec 0.1, 74.2 |
| 機密情報を扱わない | 患者名、個人情報、APIキー、実在医院名は持たない | spec 74.10 |
| 単価・購入履歴は組織機密 | 組織外には表示しない、外部共有しない | spec 73-74 |
| 取り込み元ファイル長期保存しない | 購入履歴の元ファイル本体はDBに保存しない | spec 74.2, 74.8 |
| 自動発注・自動補充は作らない | 外部発注送信、メール送信、PDF生成、自動補充は範囲外 | spec 9, 74.9 |
| 操作元は `StockMovement.sourceType` で識別 | 在庫変更操作はすべて `stock_movements` に履歴を残す | spec 各画面 |

これらは差別化軸の「無駄削減」を実現するためにも維持すべき制約です。
「自動で在庫を減らす／自動で発注する」方向に踏み込むと、安全性・信頼性の競合優位が失われます。

## 4. 差別化軸の整理: 院内在庫管理の6つの無駄

販売トークに使える整理として、以下の6種類で無駄を分類しています。

1. **廃棄ロス**: 期限切れ・劣化で使えず捨てる商品
2. **過剰在庫（死蔵）**: 念のため買って眠る商品、年単位で動かない在庫
3. **欠品ロス**: 不足によるキャンセル・代用品・急ぎ発注の高単価
4. **探す時間**: 「どこ？」「あといくつ？」「これ何だっけ？」
5. **重複発注**: 複数スタッフで同じ商品を発注、ディーラーから二重納品
6. **属人化・引き継ぎコスト**: 古参スタッフの頭の中にしかない発注ルール

現状アプリは 1 と 4 の一部（期限ロット一覧、保管場所表示）まで可視化済み。
ここから「数値とアクションに変える」のが次フェーズです。

## 5. 強化機能候補 S-1〜S-10

各機能はすべて「現状実装の上に積む」ことを前提にしています。

### S-1. 適正在庫レコメンド（推奨 minStock を表示）

- 概要: 直近 30/60/90 日の出庫数量から、商品ごとの推奨 minStock を計算して表示
- 表示場所候補:
  - 商品一覧の「最低在庫」セル横に推奨値
  - 商品詳細の在庫情報セクション
  - 一括整備画面（購入履歴登録商品の整備で活用）
- データ源: `StockMovement` の `movementType = OUT` を集計
- DB スキーマ追加: 不要
- 採用方式: 「推奨値」を表示するだけ、採用は人が確定（仕様 0.1 「在庫は人が確定」を維持）

### S-2. 期限切れ予防アラート + FIFO 出庫支援

- 概要:
  - 期限切れ寸前のロット金額を月次でホームに表示
  - バーコード出入庫・クイック出庫で、対象商品にロット複数がある時に「最古ロット優先」表示
- 表示場所候補:
  - ホームの要対応セクション（「廃棄予防可能 X 円」）
  - バーコード出入庫の商品特定後画面
- データ源: 既存 `StockLot` モデルと `Product.standardPrice`
- DB スキーマ追加: 不要

### S-3. 死蔵在庫レポート（N日間出庫なし）

- 概要: 過去 90 日間出庫が無く、在庫1以上の商品を一覧表示。標準価格 × 数量で滞留金額を表示
- 画面候補: 新規ページ `/stock-lots/dormant` または `/inventory/dormant`
- データ源: `StockMovement.createdAt` を商品ごとに集約
- DB スキーマ追加: 不要

### S-4. 発注済み・納品待ちの可視化（重複発注の根絶）

- 概要:
  - 不足在庫一覧と商品一覧で「発注済（納品待ち）N個」を表示
  - 発注候補の追加ボタン横に「同じ商品が発注済」の警告
  - 商品詳細上部に未納の発注を表示
- データ源: `OrderRequest.status = ORDERED` で `receivedAt = null` の集計
- DB スキーマ追加: 不要
- **既存データだけで実装でき、デモ素材としても作りやすい。最優先候補**

### S-5. リードタイム学習（minStock 推奨の補正）

- 概要: `OrderRequest.orderedAt` と `receivedAt` の差から、発注先ごとの平均納品リードタイムを計算
- 表示場所: 商品詳細、発注候補画面、S-1 推奨値の算出式に組み込む
- DB スキーマ追加: 不要

### S-6. ABC分析（使用頻度別ランク）

- 概要: 過去 90 日の出庫数量で商品をランク分け（A: 上位10%、B: 中位、C: 低使用）
- 用途:
  - 棚卸セッションの確認頻度設計（A は毎月、C は半年）
  - マスタ整備の優先順位（A 商品の minStock とバーコード優先整備）
- データ源: `StockMovement` 集計のみ
- DB スキーマ追加: 不要

### S-7. 異常出庫・異常消費の検知

- 概要: 商品ごとの過去 N 日平均と当日の出庫数量を比較し、3倍以上の出庫を「異常」としてホームに通知
- 異常閾値は組織で調整可能にする
- DB スキーマ追加: 組織別の閾値設定を持つなら新規モデル要

### S-8. 月次「無駄削減金額」ダッシュボード

- 概要: 各 S-X 機能の効果を金額で表示するホームのカード
  - 廃棄予防可能額（S-2 由来）
  - 重複発注を防いだ件数 × 平均単価（S-4 由来）
  - 推奨 minStock 採用で削減できた在庫金額（S-1 由来）
  - 死蔵在庫として認識された金額（S-3 由来）
- 設計上の注意: 「組織機密扱いで金額表示しない」モードを用意する

### S-9. 商品マスタ重複候補検知

- 概要: 既存マスタ内で「メーカー名 + 商品名 + 規格が似ている」商品を重複候補として一覧表示
- ADMIN が確認して「同一として統合」または「別商品として確認済」を選ぶ
- 統合時は片方を `isActive = false` にし、`importSource` や `productSuppliers` を引き継ぐ
- 類似度ロジックは `src/lib/imports/purchase-history-import.ts` の `similarityScore` を再利用可能
- MVP では「候補表示まで、統合は手動」が安全

### S-10. 朝のダイジェスト通知（メール / LINE）

- 概要: 毎朝 7:00 に当日の要対応一覧を ADMIN とスタッフ別にメール送信
- 仕様 9 節で現状「メール送信は作らない」となっているため、実装時は仕様改訂が必要
- ステップ実装: まず Web 内通知（ホームのバナー）→ メール送信は要件確定後

## 6. 販売トーク用に3本柱にまとめると

S-1〜S-10 を販売説明用に3本に集約すると伝わりやすい:

1. **データで「適正在庫」を出すから、買いすぎない・切らさない**
   → S-1（推奨 minStock）+ S-5（リードタイム）+ S-6（ABC分析）
2. **期限切れ廃棄と死蔵在庫を金額で見える化、捨てる前に動ける**
   → S-2（FIFO 支援）+ S-3（死蔵）+ S-8（金額ダッシュボード）
3. **スタッフが入れ替わっても「誰が頼んだ・あといくつ来る」が分かる**
   → S-4（発注済可視化）+ S-7（異常検知）+ S-10（朝ダイジェスト）

## 8. 第一着手の推奨: S-4 発注済可視化

最も早く価値が出るのは S-4 です。既存データだけで実装でき、デモシナリオに直接組み込めます。

### 8.1 仕様メモ（実装前に `docs/spec.md` に反映）

新規節として、例えば `## 75. フェーズ4-A 発注済可視化と重複発注防止` を追加し、以下を記述する想定:

- 不足在庫一覧 (`/shortage`) の各商品行に「発注済 N個（最終発注日: YYYY-MM-DD）」を表示する
- 商品一覧 (`/products`) の各商品行に「発注済 N個」を表示する
- 商品詳細 (`/products/[productId]`) の上部に未納の発注を一覧表示する
- 不足在庫から発注候補に追加するボタン横に、同じ商品が既に発注済の場合は警告を表示する
- 「発注済」の定義: `OrderRequest.status = ORDERED` かつ `OrderRequest.receivedAt = null` で集計
- 集計対象はログインユーザーの所属組織・所属クリニックに限定する
- 在庫数は変更しない、発注候補の状態は変更しない（人が判断する）

## 9. 設計・運用上の注意点（全機能共通）

- すべての推奨値・アラート機能は「人が確認してボタンで採用」原則を維持する
- 廃棄防止金額・死蔵在庫金額・年間使用金額は組織機密。本部ダッシュボードの権限と「金額表示しない」モードを用意する
- 推奨値計算の根拠（過去90日間出庫の合計、平均、リードタイム）は商品詳細で開示する
- ABC分析・死蔵在庫の集計は組織×医院単位で。法人本部からは医院別とサマリの切り替えを用意
- 異常検知は閾値の調整可能性が肝。固定値だとシーズン物・新規導入物で誤検知
- 通知系（S-10）は仕様 9 節で未実装としているメール送信に踏み込むため、仕様書 9 節の更新を伴う

## 10. 避けるべき方向（差別化を曇らせる領域）

販売の差別化として手を出さない方が良い領域:

- ディーラー API 連携: 各社の仕様調整に半年〜1年掛かり、MVPの差別化を曇らせる
- 患者・治療単価・保険請求への接続: 法規制（特に診療情報）に踏み込むと販売スピードが落ちる
- アプリ内 PDF 生成・自動メール発注: 仕様で明示的に「やらない」としており、競合と機能比較に走るリスク
- 完全自動補充・AI 発注: 「人が確定する」原則と矛盾するので、入れるなら別商品で

## 12. 不明点・確認したいこと

実装に入る前に確認しておきたい点。これらは仕様書には未記載のため、ユーザー判断が必要です。

- S-1 の推奨 minStock 計算式は「過去90日の月間平均使用 × リードタイム日数 / 30 + 安全在庫」でよいか。安全在庫の決め方は固定値か、組織設定か
- S-3 の死蔵在庫の閾値は「90日間出庫なし」で良いか。期限切れ間近の在庫もここに含めるか別扱いか
- S-7 異常検知の閾値は「過去30日平均の3倍以上」で良いか。商品ごとに調整できる必要があるか
- S-8 金額ダッシュボードを ADMIN のみ閲覧可能とするか、スタッフも見られるようにするか
- S-10 メール送信を導入する場合の送信基盤（SendGrid / Resend / SMTP）の選定方針

これらは次フェーズ仕様化の議論ポイントです。実装前にユーザーに確認してください。
