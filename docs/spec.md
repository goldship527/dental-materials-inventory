## 0. この仕様書の位置づけ

この文書は、スタッフが日常操作を覚えるためのマニュアルではない。

この仕様書では、一般歯科材料の在庫管理業務について、次のような業務上の要素を整理する。

- 何を管理対象にするか
- どの画面や操作が必要か
- どの情報を表示・記録するか
- どの操作で在庫や履歴が変わるか
- どの機能を今は作らないか
- 安全上、どの機能を慎重に扱うか

スタッフ向けの操作説明は `docs/user-manual.md` に分ける。

開発・管理用の起動手順、進捗整理、環境メモは `docs/project-manual.md` に分ける。

開発作業の時系列記録は `docs/dev-log.md` に分ける。

外出先で画面を見せるための公開デモ手順は `docs/demo-deploy.md` に分ける。

この仕様書はフェーズごとの検討履歴も残しているため、第三者レビューではまず以下の「現状サマリー」と「レビューしてほしい観点」を読む。その後、必要に応じて各フェーズの詳細仕様を確認する。

## 0.2 第三者レビューで見てほしい観点

第三者レビューでは、実装の細かいコードだけでなく、次の観点を確認してほしい。

業務適合性:

- 歯科医院の日常的な材料管理として、在庫確認、不足確認、出庫、入庫、棚卸、発注候補の流れが自然か
- 実際のスタッフが迷いやすい箇所、押し間違えやすい箇所がないか
- バーコード出入庫の操作手順が現場で使える粒度になっているか
- 初期導入時に、商品・発注先・バーコード・最低在庫を登録する負担が大きすぎないか

販売前の不足:

- 有料提供前に必須となる管理機能、サポート機能、運用ルールが何か
- データ保護、バックアップ、障害対応、退職者アカウント停止などの運用リスクは十分に洗い出せているか
- 1医院向け、複数医院向け、代理店・本部向けのどの売り方が現実的か

設計・保守性:

- DBモデル、履歴設計、在庫更新処理が今後の拡張に耐えられるか
- `StockMovement.sourceType` による操作元管理で十分か、専用Enum化や関連テーブル化が必要か
- 商品写真、バーコード、棚卸、発注候補の境界が保守しやすいか
- Vercel + Supabase 構成で小規模販売を始める場合の懸念は何か

## 1. 目的

一般歯科材料・消耗品・備品について、日々の在庫確認と補充判断を簡単にするための在庫管理アプリを作る。

将来の発注管理や複数クリニック対応を見据えつつ、まずは「在庫が見える」「不足が分かる」「簡単に数量を増減できる」「バーコードで商品を特定できる」「棚卸結果を履歴として残せる」状態を作る。

## 2. MVP

現状のMVPは「在庫見える化、発注候補確認、発注済み記録、簡易納品確認、バーコード出入庫、期限ロット確認、棚卸セッションまでの試用体験」です。

シードデータを使い、次の流れを一連のデモとして見せられることを目指す。

1. ログインする
2. ホームで不足件数を見る
3. 在庫一覧で全商品を見る
4. 在庫数を理由メモ付きで直接編集する
5. クイック出庫で `+1 / -1` する
6. 不足在庫一覧を見る
7. 不足在庫一覧をブラウザ印刷する
8. 不足商品を発注候補に追加する
9. 発注候補の発注先、数量、状態、メモを確認する
10. 発注書下書きをブラウザ印刷またはPDF保存し、人が発注先へ送付する
11. 送付後に発注候補を発注済みにし、送付方法や送付メモを記録し、納品時に必要に応じて在庫へ反映する
12. 商品、発注先、バーコード情報を確認・最小編集する
13. バーコードを読み取り、商品を確認して入庫または出庫を確定する
14. 棚卸セッションを開始し、実在庫を途中保存し、差異を確認して確定する
15. 入出庫履歴、期限ロット一覧、棚卸履歴、バーコード読取履歴を確認し、必要に応じて入出庫履歴をCSV出力する

現状MVPとして、以下を満たすことを完了条件とする。

- `test@example.com / password` でログインできる
- ログアウトできる
- パスワード変更画面を表示できる

- ホーム画面に日常操作、在庫登録数、不足件数、クイック出庫カード数、発注候補、直近更新などの概要が表示される
- 共通ナビゲーションで主要画面を移動できる
- 在庫一覧で50件程度の商品在庫が表示される
- 在庫数の直接編集時に理由メモを必須にできる
- クイック出庫で20品程度の `+1 / -1` ができる
- クイック出庫の商品カードをカテゴリ別に絞り込める
- 不足在庫一覧には `現在庫 < 最低在庫` の商品のみ表示される
- 不足在庫一覧をA4縦想定で印刷できる
- 不足在庫から発注候補を作成できる
- 発注候補の数量、状態、備考、発注先を変更できる
- 発注先ごとに発注書下書きをブラウザ印刷またはPDF保存できる
- 発注済み候補を記録し、発注済み日時、送付方法、送付メモ、先方対応メモを確認できる

- 発注先マスタを検索、閲覧、詳細確認、新規作成、編集できる

- 商品に複数バーコードを紐づけられる
- バーコード検索で商品一致、複数候補、未登録を判定できる
- バーコード読取履歴を保存し、未対応履歴を整理できる
- バーコード出入庫で、商品が1件に特定できた場合だけ、数量と理由を確認して入庫・出庫できる
- 棚卸セッションで実在庫入力、途中保存、スキップ、破棄、差異表示、確定、履歴詳細確認ができる
- 在庫変更は `stock_movements` に履歴として残る
- 入出庫履歴を検索、区分、操作元、期間で絞り込み、CSV出力できる
- 取り消し可能な履歴は、現在庫が後続操作で変わっていない場合に取り消せる
- 期限ロット一覧で、期限切れと30日以内のロット別在庫を確認できる
- 公開デモでは、Vercel + Supabase PostgreSQL 構成で架空データを表示できる

## 7. 画面仕様

### 共通ナビゲーション

- ログイン後の主要画面で、画面上部に共通ナビゲーションを表示する
- 通常業務メニューとして、ホーム、在庫一覧、出庫、バーコード、不足一覧、発注候補、履歴、棚卸、商品、発注先へ移動できる
- ADMINには管理モードへの導線を表示し、管理モードでは初期設定、取込確認、ユーザー管理、監査ログへ移動できる
- マニュアル、アカウント、ログアウトは通常業務や管理操作とは分けた補助リンクとして表示する
- 画面幅が狭い場合は横スクロールで表示する
- ログアウト導線を表示する

### ログイン

- メールアドレスとパスワードでログインする
- パスワード入力欄には、入力内容を一時的に確認できる表示/非表示ボタンを置く
- 成功したらホームへ移動する
- 公開デモ用ログイン情報は環境変数で管理し、仕様書やGitには記載しない

### ホーム

- 日常操作として、クイック出庫、不足在庫、発注候補、バーコード出入庫へ移動できる大きな導線を表示する
- 在庫登録数、総在庫数、不足件数、在庫0件数、クイック出庫カード数、発注候補件数を表示する
- 不足在庫の簡易推移を表示する
- 在庫0、不足在庫、未確認の発注候補、期限ロットを要対応として表示する
- 管理・確認メニューとして、在庫一覧、商品マスタ、発注先マスタ、バーコード検索、入出庫履歴、期限ロット一覧、棚卸、初期設定へ移動できる

### 在庫一覧

- 商品名、カテゴリ、現在庫、最低在庫、ステータス、保管場所を表示する
- 商品名、商品コード、JANコードで検索できる
- カテゴリと不足のみで絞り込みできる
- 数量編集時は新数量と理由メモを入力する
- 編集は `StockMovement.movementType = ADJUST`、`sourceType = MANUAL` として履歴に残す
- 更新中表示、成功メッセージ、エラーメッセージを表示する

### クイック出庫

- 1カードを1商品として表示する
- 商品名、カテゴリ、現在庫数、最低在庫、保管場所を表示する
- 商品写真がある場合はサムネイルを表示する
- `+1`、`-1` で入出庫する
- カテゴリタブでカードを絞り込める
- 操作履歴を画面下部に表示する
- 操作は `StockMovement.sourceType = QUICK_CARD` として履歴に残す
- 理由メモは毎回入力させず、通常使用のクイック操作として自動記録する

### 不足在庫一覧

- `stock_items.quantity < COALESCE(stock_items.minStock, products.defaultMinStock)` の商品を表示する
- 商品名、現在庫、最低在庫、不足数、発注先を表示する
- 商品名、商品コード、JANコード、カテゴリ、メーカー、発注先で検索できる
- 不足商品を発注候補に追加できる
- 印刷ボタンで `window.print()` を呼ぶ
- 印刷時は操作UIを非表示にし、発行日時、確認者欄、発注確認欄、備考欄を表示する

### 発注候補

- 不足在庫一覧から作成された発注候補を表示する
- 商品名、発注先、現在庫、最低在庫、推奨数量、発注数量、状態、備考を確認できる
- 商品名、商品コード、カテゴリ、発注先、備考で検索できる
- 状態は `未確認`、`確認済み`、`取り消し`、`発注済み` として扱う
- 発注数量、状態、備考を更新できる
- 発注数量は直接入力に加えて、画面上の `-` / `+` ボタンで1ずつ増減できる。ただし保存は更新ボタンを押した場合だけ行う
- 発注数量の最小値は1とし、発注をやめる場合は数量0ではなく状態を `取り消し` にする
- `取り消し` は赤字系の表示にして、発注対象外であることを分かりやすくする
- 商品に複数の取扱発注先がある場合は、未確認または確認済みの発注候補で今回使う発注先を選べる
- 実際にFAX、メール、電話、LINEで発注した後は、発注先ごとに `発注済み` へ変更できる
- `発注済み` にする時、送付方法、送付メモ、先方対応メモを記録できる。ただし送付方法は `FAX`、`メール`、`電話`、`LINE` の4種類から選択する
- `発注済み` は発注書下書きの対象外にし、未対応の発注候補と区別する
- `発注済み` の候補には納品確認フォームを表示し、必要に応じて在庫へ入庫反映できる
- 納品確認済みの候補は、発注済み状態のまま納品確認を取り消せる
- 発注先ごとにグルーピングして表示する
- 発注書下書きは発注先ごとに改ページし、A4縦想定でブラウザ印刷またはPDF保存できる

### 商品マスタ

- 商品一覧、検索、カテゴリ絞り込み、不足絞り込みを表示する
- 商品詳細で基本情報、在庫、発注候補、入出庫履歴、バーコード、商品写真を確認できる
- 商品編集で基本情報、主発注先、代替発注先、発注先品番、最低在庫などを編集できる
- 商品写真を1枚登録、表示、削除できる
- 商品マスタ新規作成画面で、基本情報だけを新規登録できる
- 商品マスタCSV/Excel貼り付け一括取り込みで、プレビュー確認後に商品を追加できる
- 商品削除は現時点では作らない

### 発注先マスタ

- 発注先一覧、検索、不足あり絞り込み、発注候補あり絞り込みを表示する
- 発注先詳細で連絡先、取扱商品、不足商品、発注候補を確認できる
- ADMINは発注先を新規作成し、発注先名、連絡先、備考を編集できる
- STAFFは発注先情報を閲覧できるが、作成・編集はできない
- 発注先の削除は現時点では作らない

### バーコード検索

- `/barcode` でバーコードまたはGS1形式の読み取り値を入力できる
- 入力値は検索前に全角英数字、全角記号、全角スペースを半角相当へ正規化する
- `products.janCode` と `product_barcodes.barcode` を検索対象にする
- 商品1件一致、複数一致、未登録、取込サンプル一致を区別して表示する
- GS1形式からGTIN、有効期限、ロット番号、シリアル番号を可能な範囲で解析して表示する
- 読取履歴を手動保存できる
- 未登録バーコードから既存商品へ紐づける導線を表示する

### バーコード出入庫

- `/barcode/stock` でバーコードから商品を特定し、数量と理由を確認して入庫・出庫できる
- 入力欄にフォーカスがない場合でも、画面全体でスキャナー入力を捕捉する
- 商品未選択時にスキャナー入力を捕捉した場合は、商品バーコードとして自動検索する
- 商品が1件に特定された場合は、担当者バーコード欄へ自動でフォーカスを移す
- 商品特定後にスキャナー入力を捕捉した場合は、担当者バーコードとして入力する
- 商品バーコードと担当者バーコードは、照合前に全角英数字、全角ハイフン、余分な前後空白を正規化する
- 商品が1件に特定できない場合は在庫操作フォームを表示しない
- 読み取っただけでは在庫を変更しない
- 出庫理由は `使用`、`その他` とする
- 入庫理由は `納品`、`返品戻り`、`その他` とする
- `棚卸調整` は棚卸セッション経由の調整理由として扱い、バーコード出入庫では選択できない
- 補足メモは任意とし、個人情報や秘密情報を書かない注意を表示する
- 操作は `StockMovement.sourceType = BARCODE_STOCK` として履歴に残す

### 入出庫履歴

- 在庫変更履歴を一覧表示する
- 商品名、商品コード、カテゴリ、理由メモ、操作者で検索できる
- 操作種別と操作元で絞り込みできる
- 開始日と終了日で期間を絞り込みできる
- 現在の検索条件、操作種別、操作元、期間を反映してCSV出力できる
- CSVはExcelで開きやすいようUTF-8 BOM付きにする
- CSV出力項目は、日時、商品名、商品コード、カテゴリ、区分、増減数、変更前数量、変更後数量、操作元、理由メモ、操作者、ロット番号、有効期限とする
- 画面表示は条件一致のうち最新100件まで、CSV出力は条件一致のうち最大5000件までとする
- `MANUAL`、`QUICK_CARD`、`BARCODE_STOCK`、`ORDER_RECEIPT`、`ORDER_RECEIPT_REVERT`、`STOCKTAKE`、`STOCKTAKE_SESSION`、`REVERT` を操作元として扱う
- 後続操作で現在庫が変わっていない履歴は取り消しできる
- 棚卸セッション由来の履歴は取り消し対象外とする
- 納品確認由来の履歴は入出庫履歴画面の汎用取り消し対象外とし、発注候補画面の納品確認取り消しで扱う

### 期限ロット一覧

- `/stock-lots` で数量が1以上のロット別在庫を一覧表示する
- 商品名、商品コード、JAN、カテゴリ、ロット番号、有効期限、数量、更新日時を表示する
- 期限切れ、30日以内、期限あり、期限未登録を状態として表示する
- 初期表示は、期限切れと30日以内の「要確認」ロットに絞り込む
- 要確認、期限切れ、30日以内、すべてのフィルターを切り替えられる
- 商品名、商品コード、JAN、カテゴリ、メーカー、ロット番号、有効期限文字列で検索できる
- 商品名から商品詳細へ移動できる
- 期限ロット一覧では在庫数を変更しない

### 棚卸セッション

- `/stocktake/sessions` で進行中セッションと履歴を表示する
- `/stocktake/sessions/new` で棚卸セッションを開始する
- 開始時に対象商品のスナップショットを `StocktakeSessionItem` として作成する
- セッション詳細で商品検索、カテゴリ、状態タブ、バーコード入力により対象商品を探せる
- セッション詳細の標準表示は「すべて」とし、実在庫保存後も同じ行を表示し続ける
- 未入力だけを確認したい場合は「未入力のみ」タブで絞り込む
- 入力済み、スキップ済み、直近操作行は行背景と状態ラベルで見分けられるようにする
- 実在庫を入力し、差異を保存できる
- 未入力、入力済み、スキップを区別する
- セッションは破棄または確定できる
- 確定時に在庫数を更新し、`StockMovement.sourceType = STOCKTAKE_SESSION` として履歴を残す
- 履歴詳細で差異あり、差異なし、未入力を確認できる

### アカウント

- ログイン中ユーザーのパスワードを変更できる
- 現在のパスワード、新しいパスワード、確認入力を使う
- 各パスワード入力欄には、入力内容を一時的に確認できる表示/非表示ボタンを置く
- ADMINはユーザー追加、無効化、パスワード再設定を行える
- ログイン失敗が続いた場合は一時ロックする

### 管理モード

- ADMINだけが初期設定、取込確認、ユーザー管理、監査ログへ移動できる
- ユーザー管理では、同じ組織内のユーザー作成、無効化、パスワード再設定を行える
- 監査ログでは、管理者操作や重要なマスタ変更を確認できる

### 22.1 商品編集画面のバーコード管理

`/products/[productId]/edit` にバーコード管理セクションを追加する。

できること:

現時点では、実運用への影響が小さい順に進めるなら、バーコードスキャナー実機確認、棚卸セッション強化、発注書単位ワークフローの順が自然である。

## 25. フェーズ3-E バーコードスキャナー実機確認とGS1読み取り補助

フェーズ3-Eでは、到着したバーコードスキャナーをキーボード入力として使う前提で、`/barcode` の実機運用を確認しやすくする。

目的:

- スキャナーで読み取った値を、すぐ `/barcode` の入力欄で検索できるようにする
- 医療機器・医療材料で使われるGS1形式の読み取り値から、商品コード部分を検索候補として扱う
- 読み取りは商品確認とマスタ整備の補助に留め、在庫数や発注候補は自動変更しない

### 25.1 GS1読み取り値の扱い

GS1形式のバーコードでは、商品コードだけでなく、有効期限、ロット番号、シリアル番号などが含まれる場合がある。

フェーズ3-Eでは、アプリケーション識別子 `01` のGTINを読み取り、検索候補として使う。

検索候補:

- 読み取り値そのもの
- `01` から取り出した14桁GTIN
- 14桁GTINが先頭0の場合、先頭0を外した13桁JAN候補

判断:

- 期限、ロット番号、シリアル番号は、このフェーズでは保存しない
- 期限管理、ロット管理、トレーサビリティは別フェーズで検討する
- 未登録時に商品へ紐づける場合は、GS1全体ではなく商品コード候補を使う

### 25.2 実機スキャナー向けUI

`/barcode` の入力欄は、画面表示時と検索後にフォーカスされる。

入力欄では数字だけでなく、GS1読み取り値に含まれる括弧や制御文字相当の値も扱えるようにする。

検索結果では、GS1からGTINを取り出せた場合に、取り出した商品コードとJAN候補を表示する。

### 25.3 フェーズ3-Eでまだ作らないもの

- バーコード読み取りによる在庫の自動入出庫
- バーコード読み取りによる棚卸自動確定
- 有効期限、ロット番号、シリアル番号の保存
- 医療機器データベースや商品マスタCSVの自動取り込み
- 外部発注送信
- メール送信
- 発注書PDF生成
- 納品確認

### 25.4 医療機器・医療材料リストの扱い

医療機器の一般的名称やJMDNコードの一覧、医療機器データベース、医療材料データベースは存在する。

ただし、商品名、メーカー、JAN/GTINなどを含む商品マスタは、利用条件やダウンロード条件を確認する必要がある。

このプロジェクトでは、実在の商品データをseedやコードに直接入れない。将来取り込む場合は、正規に取得したCSV等をローカル環境で確認し、取り込み前に対象項目、重複時の扱い、更新時の上書きルールを決める。

## 26. フェーズ3-F 医療機器データ取り込みプレビューとバーコードテスト印刷

### 26.1 実データの扱い

## 27. フェーズ3-G スキャナー日時付与の扱い

当初はスマホ連携スキャナーアプリが読み取り値に時刻や日付を付ける可能性を考慮していた。

2026-05-24時点では、運用方針を「スキャナーは読み取り値に日付・時刻を付けない」に変更する。

### 27.1 解析方針

`/barcode` の入力値から、JAN、GTIN、GS1形式の商品コード候補を抽出する。

読み取り値に含まれる時刻や日付は、商品検索や読取履歴の管理情報として扱わない。

検索候補:

- 入力値そのもの
- 入力値から抽出した13桁JAN
- GS1形式の場合は、GTINとJAN候補

### 27.2 画面表示

`/barcode` では、検索値に加えて次を表示する。

- 抽出JAN
- GS1商品コード
- GS1付帯情報がある場合の有効期限、ロット番号、シリアル番号

読み取り日時は表示しない。商品マスタに登録済みの商品があれば、従来どおり商品情報を表示する。

商品マスタに未登録でも、ローカルの医療機器データ取り込みサンプルにJANが一致する場合は、「取込サンプルに一致」として製品名、メーカー、一般的名称、JMDNコードを表示する。

### 27.3 判断

- 入出庫や読取履歴の時刻は、アプリ側の保存時刻または確定時刻を正とする
- スキャナーの読み取り値には日付・時刻を付けない
- 読み取りだけでは在庫数、発注候補、在庫履歴を変更しない

## 28. フェーズ3-H 取込サンプルからのテスト商品追加

フェーズ3-Hでは、バーコード実機テストをしやすくするため、取込サンプルに一致したJANをローカル検証用の商品として1件ずつ在庫へ追加できるようにする。

目的:

- 実在包装がなくても、サンプルJANを読んだ後に通常の商品検索まで確認できるようにする

- 一括登録ではなく、利用者が確認した1件だけを追加する

### 28.1 追加されるデータ

`/barcode` で商品マスタ未登録、かつ取込サンプルに一致した場合、「テスト商品として在庫に追加」ボタンを表示する。

押した場合、次を作成する。

- `products`
  - 商品名: サンプルの製品名
  - JANコード: サンプルのJANコード
  - 商品コード: `TEST-` + JAN末尾8桁
  - カテゴリ: サンプルの一般的名称
  - メーカー: サンプルの製造販売業者名
  - 発注単位: サンプルの包装単位
  - 標準最低在庫: 1
  - 備考: 取込サンプルから作成したローカル検証用であること、元ファイル、元シート、元行、JMDNコードなど
- `stock_items`
  - 対象クリニック: 現在ログイン中の所属クリニック
  - 現在庫: 0
  - 最低在庫: 1
  - 保管場所: 検証用
- `product_barcodes`
  - JANコードを代表バーコードとして登録

### 28.2 重複時の扱い

同じJANコードが既に同じ組織の商品マスタまたはバーコードに存在する場合、新しい商品は作らない。

## 31. フェーズ3-J 未対応バーコード整理

フェーズ3-Jでは、保存済みのバーコード読取履歴から、商品マスタ整備が必要な読み取りを1件ずつ整理できるようにする。

目的:

- 未一致バーコードを後から確認できるようにする
- 商品候補が複数ある読み取りを、商品確定前の確認対象として分けて扱う
- 取込サンプル一致の読み取りを、ローカル検証用商品として昇格できるようにする
- 既存商品への紐づけ、商品化、無視のいずれかに振り分けられるようにする
- 整理操作では、在庫数、発注候補、在庫変更履歴を変更しない方針を維持する

### 31.1 対応状態

`BarcodeScanLog` に対応状態を持たせる。

主な項目:

- `resolveStatus`
  - `UNRESOLVED`: 未対応
  - `RESOLVED_LINKED`: 既存商品へ紐づけ済み
  - `RESOLVED_PROMOTED`: 取込サンプルから商品化済み
  - `RESOLVED_IGNORED`: 無視
- `resolvedAt`
- `resolvedByUserId`
- `resolvedNote`

既存ログは `UNRESOLVED` として扱う。

### 31.2 対象画面

`/barcode/scans/unresolved` に未対応バーコード整理画面を追加する。

表示対象:

- `BarcodeScanLog.matchType = NO_MATCH`
- `BarcodeScanLog.matchType = PRODUCT_MULTI`
- `BarcodeScanLog.matchType = SAMPLE`
- `BarcodeScanLog.resolveStatus = UNRESOLVED`

表示件数は直近100件までとする。

表示する内容:

- 保存日時
- 判定種別
- 元の読み取り値
- 抽出バーコード
- 抽出JAN
- 抽出GTIN
- 取込サンプル情報
- 操作者
- バーコード検索画面への導線
- 商品マスタで紐づけ先を探す導線
- 取込サンプルから商品化する操作
- 無視する操作

`PRODUCT` は商品が確定済みのため、未対応整理画面には含めない。

### 31.3 操作

未対応整理画面では、次の操作を行える。

既存商品へ紐づけ:

- `NO_MATCH`、`PRODUCT_MULTI`、`SAMPLE` のすべてで表示する
- 既存の `/products?attachBarcode=...` 導線を使い、商品編集画面でバーコード追加を行う
- バーコード追加が成功した場合、現在クリニック内で同じバーコードに一致する未対応ログを `RESOLVED_LINKED` にする

取込サンプルから昇格:

- `SAMPLE` の場合だけ表示する
- 取込サンプル情報からローカル検証用商品を作成する
- 作成後、対象ログを `RESOLVED_PROMOTED` にする
- `NO_MATCH` や `PRODUCT_MULTI` ではサーバー側でも拒否する

無視:

- `NO_MATCH`、`PRODUCT_MULTI`、`SAMPLE` のすべてで表示する
- 任意メモとともに `RESOLVED_IGNORED` にする
- 今フェーズでは無視の取り消しUIは作らない

### 31.4 判断

- 履歴の削除は行わない
- 既存商品への紐づけは、既存の `/products?attachBarcode=...` と商品編集画面の導線を使う
- `PRODUCT_MULTI` は読取履歴の時点では商品を確定しない
- 一覧取得はログインユーザーの現在クリニックに限定する
- 対応更新時も、画面から渡されたIDだけを信用せず、サーバー側で現在クリニック、対応状態、判定種別を確認する
- 旧 `/barcode/scans/unmatched` は `/barcode/scans/unresolved` へ誘導する

### 31.5 フェーズ3-Jでまだ作らないもの

- 未一致履歴の一括整理ワークフロー
- 読取履歴自体の編集、削除
- 無視の取り消しUI
- 未一致履歴からの商品新規作成
- 読取履歴からの在庫自動入出庫
- 読取履歴からの棚卸自動確定
- CSVエクスポート

## 32. フェーズ3-K 棚卸セッションのデータ設計

フェーズ3-Kでは、棚卸を途中保存できる「セッション」として扱うためのデータ土台を追加する。

目的:

- 半日以上かかる棚卸でも、途中中断や担当者交代に対応できるようにする
- 棚卸開始時のシステム在庫、入力済み数量、差異、状態を商品ごとに保持する
- 確定前の棚卸入力と、確定後の在庫変更履歴を分けて管理する
- スキャナーを使った棚卸入力に後続フェーズでつなげられるようにする

### 32.1 想定操作

棚卸セッションでは、後続フェーズで次の操作を想定する。

- 対象クリニックで棚卸セッションを開始する
- 開始時点の在庫商品をセッション明細として作成する
- 商品ごとに実在庫数を入力し、入力済み、未入力、スキップを管理する
- メモを残しながら途中保存する
- 担当者が交代しても同じセッションを開いて入力を続ける
- 入力内容を確認して確定、または破棄する

UIと入力ロジック、削除、複数同時セッションのバリデーションは次ステップ以降で作る。

### 32.2 保存項目

親テーブルとして `StocktakeSession` を追加する。

主な項目:

- 対象クリニック
- 状態
  - `IN_PROGRESS`: 入力中
  - `COMMITTED`: 確定済み
  - `DISCARDED`: 破棄
- メモ
- 開始日時
- 確定日時
- 破棄日時
- 開始者
- 確定者
- 作成日時、更新日時

子テーブルとして `StocktakeSessionItem` を追加する。

主な項目:

- 棚卸セッション
- 商品
- 開始時点のシステム在庫数
- 入力された実在庫数
- 差異
- 状態
  - `PENDING`: 未入力
  - `COUNTED`: 入力済み
  - `SKIPPED`: スキップ
- メモ
- 入力日時
- 入力者
- 作成日時、更新日時

同じセッション内で同一商品は1明細だけ持つ。

### 32.3 確定時の挙動概要

初回のスキーマ追加時点では確定処理は実装しなかった。フェーズ3-Kの続きでは、棚卸セッションを確定したときに次の処理を行う。

- 入力済み明細の差異を確認する
- 差異がある商品について `StockMovement` に `ADJUST` として履歴を作成する
- `StockItem.quantity` を実在庫数へ更新する
- セッションの状態を `COMMITTED` にし、確定日時と確定者を保存する

棚卸セッションの入力途中では、`StockMovement` と `StockItem` は変更しない。

### 32.4 セッション開始と途中保存

フェーズ3-Kの続きとして、棚卸セッションの開始と入力途中保存を追加する。

セッション開始:

- `/stocktake/sessions` を棚卸セッションの入口にする
- `/stocktake/sessions/new` から新しいセッションを開始する
- 同じクリニックに `IN_PROGRESS` のセッションが既にある場合は、新規作成せず既存セッションへ誘導する
- 開始時に現在の在庫一覧と同じ対象、つまり `stock_items.isUsed = true` かつ有効な商品を `StocktakeSessionItem` として一括作成する
- 明細の `expectedQuantity` には開始時点の `StockItem.quantity` を保存する

途中保存:

- 明示的な「途中保存」ボタンは置かない
- 実在庫数を1商品ずつ入力した時点で、明細単位で保存する
- 保存時は `countedQuantity`、`diff`、`status = COUNTED`、`countedAt`、`countedByUserId`、任意メモを更新する
- スキップ時は `status = SKIPPED`、`countedAt`、`countedByUserId`、任意メモを保存する
- 入力途中では `StockItem.quantity` と `StockMovement` は変更しない

### 32.5 スキャナー入力導線

進行中編集画面 `/stocktake/sessions/[sessionId]` の上部にスキャナー入力欄を常駐させる。

- JAN、GTIN、GS1は既存の `analyzeBarcodeInput` と同じ解析ルールで扱う
- 一致する商品が1件だけの場合、その明細へ自動スクロールし、実在庫入力欄へフォーカスする
- 一致しない場合、または複数一致する場合は警告だけを表示し、在庫数や棚卸明細は変更しない
- 保存時は画面上の値だけを信用せず、サーバー側でログインユーザーの現在クリニック、対象セッション、入力中状態を確認する

### 32.6 破棄と確定

進行中セッションは確認モーダルを経由して破棄できる。

- 破棄時は `StocktakeSession.status = DISCARDED` と `discardedAt` を更新する
- 破棄しても `StockItem.quantity` と `StockMovement` は変更しない
- 確定ボタンはこのステップでは無効表示にし、確定処理は次ステップで実装する

履歴詳細、確定時の在庫更新、確定済みセッションの詳細表示は次ステップ以降で扱う。

### 32.7 セッション確定と履歴詳細

進行中の棚卸セッションは、確認モーダルを経由して確定できる。

確定確認:

- 画面では「差異あり件数」と「未入力件数」を表示してから確定する
- 利用者が確定を選ぶまで、サーバー側の確定処理は実行しない
- 未入力の明細は在庫更新対象にしない

確定処理:

- `StocktakeSession.status = IN_PROGRESS` の場合だけ確定できる
- 対象セッションが現在ログインユーザーのクリニックに属することをサーバー側で確認する
- `COUNTED` の明細だけを確定対象にする
- `PENDING` の明細は差異なしとはみなさず、在庫更新対象から除外する
- `SKIPPED` の明細も在庫更新対象から除外する
- 差異がある明細について `StockMovement` に `ADJUST` として履歴を作成する
- `StockMovement.sourceType` は `STOCKTAKE_SESSION`、`sourceId` は棚卸セッションIDにする
- `StockMovement.userId` は確定者にする
- 確定対象商品の `StockItem.quantity` を `countedQuantity` に更新する
- 確定後、セッションに `COMMITTED`、`committedAt`、`committedByUserId` を保存する

同時更新への対応:

- 確定処理はトランザクション内で行う
- セッション単位と商品単位でロックを取り、同じセッション確定や同じ商品の同時確定を避ける
- 確定対象商品の現在庫が棚卸開始時の `expectedQuantity` から変わっている場合は、既存更新を上書きせず確定を止める

履歴詳細:

- `/stocktake/sessions/[sessionId]/history` で過去セッションの概要と明細を確認する
- 概要には開始者、確定者、開始日時、確定日時、メモを表示する
- 明細は「差異あり」「差異なし」「未入力」のタブで確認できる
- 商品ごとに期待数、実数、差異、関連する入出庫履歴への導線を表示する

既存の `/stocktake` は当面残す。旧方式は単発入力として `StockMovement.sourceType = STOCKTAKE` 相当の履歴を作り、新方式は `STOCKTAKE_SESSION` としてセッション単位の履歴を作る。しばらくは二系統の棚卸履歴が並ぶ前提で扱う。

## 33. フェーズ3-L 入出庫履歴の取り消し

フェーズ3-Lでは、誤入力時の安心感を上げるため、`StockMovement` の1件単位で「この操作を取り消す」機能を追加する。

目的:

- 誤って在庫数を変更した場合に、履歴を削除せず逆向きの履歴で在庫を戻せるようにする
- どの履歴を誰がいつ取り消したかを残す
- 取り消し操作自体も `StockMovement` として記録し、監査性を保つ

### 33.1 保存項目

`StockMovement` に次の項目を追加する。

- `revertOfId`: 取り消し操作として作られた逆向き履歴の場合、元の `StockMovement.id` を保存する
- `revertedAt`: 元の履歴が取り消された時刻を保存する
- `revertedById`: 元の履歴を取り消したユーザーIDを保存する

`revertOfId` には検索用インデックスを付ける。

### 33.2 操作

`/movements` の各履歴行に、条件を満たす場合だけ「取り消す」ボタンを表示する。

取り消し可能な条件:

- 現在ログインユーザーのクリニックに属する履歴である
- 商品が有効である
- 元履歴の `revertedAt` が空である
- 元履歴の `revertOfId` が空である
- `sourceType` が `STOCKTAKE_SESSION` ではない

取り消し時は確認モーダルを必須にする。

### 33.3 取り消し処理

取り消しは `prisma.$transaction` の中で実行する。

処理内容:

- 対象履歴と対象商品のロックを取る
- `StockItem.quantity` が元履歴の `afterQuantity` と一致することを確認する
- 一致しない場合は、同じ商品が後から更新された競合として取り消しを止める
- 逆向きの `StockMovement` を `movementType = ADJUST` として作成する
- 逆向き履歴の `beforeQuantity` は元履歴の `afterQuantity`、`afterQuantity` は元履歴の `beforeQuantity` にする
- 逆向き履歴の `sourceType` は `REVERT`、`sourceId` と `revertOfId` は元履歴IDにする
- `StockItem.quantity` を元履歴の `beforeQuantity` に戻す
- 元履歴に `revertedAt` と `revertedById` を保存する

更新後は `/home`、`/inventory`、`/shortage`、`/movements` を再検証する。

### 33.4 今フェーズで作らないもの

- 履歴の削除
- 取り消し操作の取り消し
- 取り消し可能期限
- 棚卸セッション由来履歴の取り消し

`sourceType = STOCKTAKE_SESSION` の履歴を取り消すと、棚卸セッション詳細に保存された実数と現在庫が乖離する。今フェーズでは安全側に倒し、棚卸セッション由来の履歴は取り消し対象外として固定する。

## 34. フェーズ3-M パスワード変更

フェーズ3-Mでは、ログインユーザーが自分のパスワードを画面から変更できるようにする。

目的:

- 共通の開発用ログイン情報を使い続けるだけでなく、必要に応じて本人がパスワードを変更できるようにする
- 管理者によるリセットではなく、ログイン中ユーザー本人の操作に限定する
- パスワードを平文保存せず、既存ログインと同じハッシュ方式で保存する

### 34.1 対象画面

`/account/password` にパスワード変更画面を追加する。

入力項目:

- 現在のパスワード
- 新しいパスワード
- 新しいパスワード（確認）

画面では成功メッセージと失敗メッセージを表示する。

共通ナビゲーションには「アカウント」導線を追加し、パスワード変更画面へ移動できるようにする。

### 34.2 入力ルール

新しいパスワードは次の条件を満たす必要がある。

- 8文字以上
- 128文字以内
- 半角英数字と記号のみ
- 確認用パスワードと一致する

### 34.3 更新処理

Server Actionでは `requireActiveClinic` ではなく `auth()` でログインユーザーを確認する。

処理内容:

- ログイン中ユーザーを取得する
- 現在のパスワードを `bcrypt.compare` で照合する
- 不一致の場合は保存せずエラーを返す
- 新しいパスワードを `bcrypt.hash`、salt rounds 12でハッシュ化する
- `User.passwordHash` を更新する

監査ログ専用テーブルは今フェーズでは追加しない。パスワード本文はログ、ドキュメント、画面以外の保存先に残さない。

### 34.4 実装・テスト方針

パスワード入力ルールは `src/lib/auth/password-schema.ts` に切り出し、Server Action とテストから同じスキーマを使う。

テストでは次を分けて確認する。

- パスワード文字数、使用可能文字、確認用入力との一致は、DBに依存しないスキーマ単体テストで確認する
- 現在パスワード照合とハッシュ保存は、パスワード変更の中核処理を直接呼び出す統合テストで確認する
- 本番処理の `bcrypt.hash(..., 12)` は維持し、テスト準備用の既存パスワードハッシュだけ軽いコストで作る

### 34.5 運用メモ

共通アカウントを複数人で使っている運用では、パスワード変更後に関係者へ新しいパスワードを共有し、必要に応じてログアウトと再ログインを行う。

### 34.6 今フェーズで作らないもの

- 他ユーザーのパスワードリセット
- 管理者向けアカウント管理画面
- アカウント削除
- 多要素認証
- 専用の監査ログテーブル

## 35. フェーズ3-N GS1付帯情報の表示

フェーズ3-Nでは、GS1形式のバーコードに含まれる有効期限、ロット番号、シリアル番号を `/barcode` の検索結果に表示する。

目的:

- GS1バーコードを読み取ったときに、商品コードだけでなく現場で確認したい付帯情報を画面上で見えるようにする
- まずは表示だけに限定し、DB保存や在庫操作にはつなげない
- 後続フェーズでロット管理や期限管理を保存する前の、解析結果確認ステップとする

### 35.1 対象AI

今フェーズでは次のアプリケーション識別子だけを解析対象にする。

- `01`: GTIN
- `17`: 有効期限
- `10`: ロット番号
- `21`: シリアル番号

### 35.2 解析ルール

括弧形式と圧縮形式の両方を扱う。

括弧形式の例:

```text
(01)04900000000009(17)270531(10)LOT123(21)SER01
```

圧縮形式の例:

```text
]C101049000000000091727053110LOT123\u001d21SER01
```

`17` は `YYMMDD` の6桁として扱う。

- 年は `2000 + YY` として解釈する
- 日が `00` の場合は、その月の月末日として扱う
- 不正な日付の場合は、Date化せず、生文字列だけを保持する

`10` と `21` は可変長として扱い、次のAIまたはFNC1区切りまでを値とする。

### 35.3 表示

`/barcode` の検索結果上部に、既存のGTIN、JANに加えて次をバッジ表示する。

- 有効期限
  - Date化できる場合は `yyyy/MM/dd`
  - Date化できない場合は `YYMMDD` の生文字列
- ロット番号
- シリアル番号

### 35.4 読取履歴への任意保存

フェーズ3-Nの続きとして、利用者が `/barcode` で「履歴に保存」を押した場合は、解析したGS1付帯情報を `BarcodeScanLog` に保存する。

保存項目:

- `lotNumber`: `10` ロット番号
- `serialNumber`: `21` シリアル番号
- `expiryDateText`: `17` 有効期限の生文字列
- `expiryDate`: `17` 有効期限をDate化できた場合の日付

検索しただけでは読取履歴は増やさない。保存時はサーバー側で `analyzeBarcodeInput` を再実行し、画面表示値ではなくサーバー側の解析結果を `BarcodeScanLog` にコピーする。

既存の読取履歴は新規カラムが `NULL` のままでよい。一覧表示では値がない場合は `-` として扱う。一覧クエリ用のindexは今フェーズでは追加しない。

`/barcode/scans` の「抽出コード」列には、既存の抽出コード、JAN、GTINに加えて、有効期限、ロット番号、シリアル番号を小さく補足表示する。SAMPLE一致時のサンプル情報とは別の補足情報として表示し、在庫や商品マスタには反映しない。

### 35.5 今フェーズで作らないもの

- 商品、在庫、入出庫履歴への自動反映
- ロット単位の在庫管理
- 使用期限アラート
- CSV出力

実在のロット番号やシリアル番号は、テスト、seed、ドキュメントに含めない。検証では架空値だけを使う。

### 36.1 追加表示

  - 確定日、差異あり件数、対象件数を表示する
  - `/stocktake/sessions` へ移動できる
- 期限が近い読取履歴件数
  - `BarcodeScanLog.expiryDate` が当日0時以降、30日後までのものを対象にする
  - 過去日の期限切れ読取はホーム件数には含めず、別の整理画面や後続フェーズで扱う
  - `resolveStatus = RESOLVED_IGNORED` の読取履歴は、整理済みとしてホーム件数から除外する
  - `/barcode/scans` へ移動できる

### 36.2 不足推移の扱い

不足件数推移は、今フェーズでは本格的な日次集計を行わない。

理由:

- `StockMovement` から過去時点の不足件数を再計算すると重くなりやすい
- 日次スナップショット用テーブルはまだ作らない
- まずはホームの情報設計と導線を軽く確認する

そのため、過去14日間の推移は「アクセス時点の現在不足件数を14日分複製した簡易表示」とする。本格的な日次履歴、増減比較、期限切れアラートは別フェーズで扱う。

ホーム画面では、スパークラインの下に「日次推移の集計は今後のフェーズで対応予定」といった注釈を表示し、水平線が実際の日次変動を表しているわけではないことを利用者に伝える。

### 36.3 UI方針

- Tailwind CSSのみで実装し、shadcn/uiは導入しない
- スパークラインは `viewBox="0 0 100 30"` 相当の軽量SVGで描画する
- 印刷時はナビゲーション、メニューカード、スパークラインなどの装飾を `print:hidden` で隠す
- 件数バッジとカードのトーンは既存ホームに合わせる

### 36.4 今フェーズで作らないもの

- DBスキーマ変更
- 日次不足件数の永続スナップショット
- `StockMovement` 履歴からの過去在庫再計算
- 期限切れアラート
- ホームからの削除、解決、在庫更新などの操作

## 37. フェーズ3-P 商品写真の1枚登録

フェーズ3-Pでは、スタッフが商品を取り違えにくくするため、商品マスタに任意で写真を1枚登録できるようにする。

目的:

- 商品名やJANだけでは判断しづらい材料を、写真で補助的に確認できるようにする
- クイック出庫や商品詳細で、小さなサムネイルを見ながら商品を選べるようにする
- 本番ストレージ設計は広げず、まずはローカル開発環境で動く最小構成にする

### 37.1 保存項目

今フェーズでは `Product` に次のnullable項目を追加する。

- `photoFileName`: 保存した写真ファイル名
- `photoMimeType`: `image/png`、`image/jpeg`、`image/webp` のいずれか
- `photoUpdatedAt`: 写真更新日時

写真は1商品につき1枚だけ登録できる。複数写真、部位別写真、添付ファイル管理は別フェーズで扱う。

### 37.2 ファイル保存

ローカル開発用の保存先は次とする。

```text
data/local/uploads/products/{productId}.{ext}
```

保存ルール:

- `public` 配下には置かない
- Git管理対象にしない
- ファイル名には利用者がアップロードした元ファイル名を使わない
- 拡張子はMIME typeから決める
- 既存写真がある場合は上書きする
- 許可形式は PNG、JPEG、WebP のみ
- 最大サイズは2MB

画像処理ライブラリは導入しない。リサイズ、圧縮、向き補正は行わず、利用者が事前に扱いやすいサイズへ調整する前提にする。

### 37.3 配信

画像は `/api/product-photos/[productId]` から配信する。

配信時の確認:

- ログイン済みであること
- 現在ログインユーザーが利用できるクリニックに属する商品であること
- 商品が有効であること
- 写真ファイル名とMIME typeがDBに保存されていること

レスポンスには `Cache-Control: private, max-age=60` を付ける。商品一覧などへの大量表示最適化は今フェーズでは行わない。

商品詳細、商品編集、クイック出庫などの画面側では、`photoUpdatedAt` をもとに `/api/product-photos/{productId}?v={timestamp}` のURLを生成する。写真を上書きした場合は `photoUpdatedAt` が変わるため、API側の短いキャッシュを残したままでも新しいURLで即時反映できる。

### 37.4 UI

商品編集画面で次を行えるようにする。

- 写真をアップロードする
- 既存写真を上書きする
- 写真だけを削除する

商品詳細画面とクイック出庫には、小さなサムネイルを表示する。写真がない場合は「写真なし」の控えめな枠を表示する。

### 37.5 実装時に確定した補足仕様

商品写真のアップロード、上書き、削除は Server Action の中核処理として実装する。画面のフォームはこの中核処理を呼び出すだけにし、保存ルールや検証ルールはサーバー側に集約する。

サーバー側の検証:

- 画像が未選択の場合は保存しない
- 2MBを超えるファイルは保存しない
- `image/png`、`image/jpeg`、`image/webp` 以外は保存しない
- 対象商品がログインユーザーの組織に属し、現在利用中クリニックの在庫対象として見えることを確認する
- 既存写真がある場合は、許可拡張子の候補ファイルを削除してから新しい写真を保存する

ブラウザ自動検証では、ローカルファイル選択ダイアログの自動操作が環境上制限されることがある。その場合は、Server Action の中核関数を直接呼び出して、PNG、JPEG、WebP、2MB超過、上書き、削除を検証する。画面側では、写真管理UI、商品詳細のサムネイル、クイック出庫のサムネイル、画像配信APIの表示を確認する。

商品写真のテストでは、PNG、JPEG、WebP の正常系と、既存写真の上書き時に古い拡張子のファイルが残らないことを確認する。

ローカル保存ファイルは `data/local/uploads/products/` に残る。検証で使った実ファイルはGit管理に含めず、必要に応じてローカルで削除する。本番運用では、ローカルディスク保存のまま使うか、オブジェクトストレージへ移すかを別フェーズで決める。

### 37.6 今フェーズで作らないもの

- 商品マスタ自体の削除
- 複数写真
- 写真の並び替え
- 画像のリサイズ、圧縮、トリミング、向き補正
- ウイルススキャン
- S3、Cloudflare R2などの本番ストレージ連携
- CDN配信

実在製品画像を検証に使う場合でも、Git管理ファイルには含めない。本番運用時の保存先、バックアップ、アクセス制御、容量管理は別フェーズで決める。

### 37.7 2026-05-21 Supabase Storage 移行

公開デモと本番候補環境では、商品写真をVercelのローカルディスクではなくSupabase Storageに保存する。

保存先:

- Supabase Storageのprivate bucketを使う
- bucket名は環境変数 `SUPABASE_STORAGE_BUCKET` から取得する。例: `product-photos`
- オブジェクトキーは `{productId}.{ext}` とし、従来のローカルファイル名と同じ規則にする

アップロード処理:

- Server Actionでログインユーザーと組織・クリニックスコープを確認する
- サーバー側でサービスロールキーを使い、Supabase StorageへPUT相当の保存を行う
- 既存写真の上書き時は、同じ商品IDの許可拡張子候補を削除してから新しいオブジェクトを保存する
- 古い拡張子のオブジェクトが残らないよう、削除→保存の順にする

配信処理:

- `/api/product-photos/[productId]` がログインユーザーと組織・クリニックスコープを確認する
- 確認後、Supabase Storageからオブジェクトをサーバー側で取得する
- レスポンスには `Cache-Control: private, max-age=60` を付けてストリーム配信する
- 画面側へSupabaseのsigned URLを直接渡さず、常にAPIプロキシ経由で配信する

ローカル開発時のフォールバック:

- 環境変数 `SUPABASE_STORAGE_BUCKET` が未設定の場合は、従来通り `data/local/uploads/` にローカル保存する
- フォールバックの分岐はストレージヘルパー1箇所に集約し、Server ActionやAPIルートへ分散させない
- `SUPABASE_STORAGE_BUCKET` を設定した場合は、`SUPABASE_URL` と `SUPABASE_SERVICE_ROLE_KEY` も必須にする

制約:

- 許可形式はPNG、JPEG、WebPのみ
- 最大サイズは2MB
- 写真は1商品1枚のみ
- 画面側の写真URLは引き続き `/api/product-photos/{productId}?v={photoUpdatedAt}` を使う

## 38. フェーズ3-Q 商品マスタの新規作成

フェーズ3-Qでは、取込サンプル経由でしか作れなかった商品マスタを、利用者が画面から1件ずつ手動登録できるようにする。

目的:

- 本番運用へ近づける前段として、業務側が必要な商品を意図的に登録できるようにする
- バーコード読み取りからの自動作成とは分け、利用者が入力内容を確認して作成する画面に限定する
- 商品作成後の在庫行、代表バーコード、写真は既存または別画面の操作に分ける

### 38.1 入力項目

必須項目:

- 商品名 `name`

任意項目:

- 商品コード `productCode`
- JANコード `janCode`
- 内部コード `internalCode`
- カテゴリ `category`
- メーカー `manufacturer`
- 規格 `specification`
- 発注単位 `orderUnit`
- 主発注先 `primarySupplierId`
- 発注先の商品コード `supplierProductCode`
- 標準価格 `standardPrice`
- 標準の最低在庫 `defaultMinStock`
- 備考 `notes`

この画面では `StockItem` は作成しない。そのため、作成直後の商品は商品マスタでは確認できるが、在庫一覧には表示されない。在庫として扱うかどうかは別の在庫追加フローで決める。

代表バーコード、商品写真、在庫数はこの画面では扱わない。代表バーコードと商品写真は商品編集画面で扱う。

### 38.2 入力ルール

- 商品名はtrim後1文字以上、100文字以内
- 商品コード、内部コード、発注先の商品コードは64文字以内
- JANコードは空欄可。入力する場合は13桁の数字
- JANコードのチェックデジット検証は今フェーズでは行わない
- 標準価格は0以上の数値
- 標準の最低在庫は0以上の整数
- 備考は1000文字以内

画面には、患者情報、実在クリニック名、会社の秘密情報、APIキーやパスワードを入力しない注意書きを表示する。

### 38.3 作成処理

Server Action `createProductAction` で作成する。

処理内容:

- `requireActiveClinic` でログインユーザーの組織とクリニックを確認する
- `organizationId` はクライアントから受け取らず、ログインコンテキストから決める
- 主発注先が指定された場合は、同じ組織内の `Supplier` かを確認してから紐づける
- `Product` だけを作成し、`StockItem`、`ProductBarcode`、写真ファイルは作成しない
- JAN重複は `@@unique([organizationId, janCode])` で防ぐ
- Prisma `P2002` を捕捉し、「同じ組織で同じJANコードの商品が既にあります」という利用者向けメッセージにする
- 成功時は新規 `Product.id` を返し、画面側で商品詳細へ遷移する

入力エラー時は、画面上部の総合メッセージに加えて、該当フィールドの下にもエラーメッセージを表示する。Zodの入力エラーは `fieldErrors` に整形し、各項目の最初のメッセージだけを表示する。JAN重複エラーも `janCode` のフィールド別エラーとして表示する。

### 38.4 商品マスタ一覧との関係

商品マスタ一覧と商品詳細は、在庫行がない商品も確認できるようにする。

在庫一覧、クイック出庫、不足在庫一覧、棚卸対象は、引き続き在庫行がある商品を中心に扱う。商品を作成しただけでは在庫数、発注候補、在庫変更履歴は変わらない。

`/products` の通常表示では「商品を新規作成」ボタンを表示する。未対応バーコードを既存商品へ紐づける `attachBarcode` モードでは、誤って新規作成へ流れないよう、このボタンは表示しない。

### 38.5 取込サンプル経由作成との分離

取込サンプル経由の `createTestProductFromSampleAction` と、新規作成画面の `createProductAction` は機能を分ける。

新規作成画面ではサンプル選択UIを出さず、利用者が入力した商品マスタ情報だけを保存する。バーコード読み取り結果から自動的に商品を作成する処理は追加しない。

### 38.6 今フェーズで作らないもの

- 商品削除
- 論理削除 `isActive = false` へ切り替えるUI
- 在庫行の同時作成
- 代表バーコードの同時登録
- 商品写真の同時登録
- 発注先の新規作成
- バーコード読み取りによる商品自動作成
- CSVインポート、CSVエクスポート
- 外部送信、メール送信、発注書PDF生成、納品確認

実在製品データはseed、コード固定値、Git管理ファイルに含めない。

## 39. 公開デモ環境の方針

外出先で第三者に画面を見せるため、認証つきの公開デモ環境を用意する。

目的:

- ローカルPCやDockerを起動していなくても、URLからデモ画面を確認できるようにする
- 本番運用ではなく、評価、説明、画面確認用の環境として扱う
- 実在データや秘密情報を入れず、架空データだけで操作感を見せる

### 39.1 推奨構成

推奨構成は次とする。

- Vercel: Next.js アプリ本体
- Supabase PostgreSQL: デモ用DB
- GitHub: Vercelへのデプロイ元

ローカル開発ではDocker上のPostgreSQLを使うが、公開デモではDockerを使わない。公開デモでは `DATABASE_URL` にSupabaseのPostgreSQL接続文字列を設定する。

### 39.2 環境変数

公開デモでは、少なくとも次の環境変数をホスティングサービス側に設定する。

- `DATABASE_URL`: Supabase PostgreSQLの接続文字列
- `AUTH_SECRET`: Auth.js用の32文字以上のランダムな秘密文字列
- `AUTH_URL`: 公開デモURL
- `DEMO_LOGIN_EMAIL`: seedで作るデモ用ログインメール
- `DEMO_LOGIN_PASSWORD`: seedで作るデモ用ログインパスワード
- `DEMO_USER_NAME`: seedで作るデモ用ユーザー表示名
- `SUPABASE_URL`: 商品写真保存に使うSupabaseプロジェクトURL
- `SUPABASE_SERVICE_ROLE_KEY`: 商品写真のサーバー側保存・取得に使う秘密値
- `SUPABASE_STORAGE_BUCKET`: 商品写真用private bucket名

接続文字列、DBパスワード、`AUTH_SECRET`、デモ用パスワード、`SUPABASE_SERVICE_ROLE_KEY` は、Git、README、チャット、スクリーンショットに含めない。

### 39.3 DB初期化

Supabase DBには、ローカルPCから `corepack pnpm db:push` と `corepack pnpm db:seed` を実行して初期化する。

`db:seed` は既存データを削除して架空データを入れ直す。デモDBはリセット可能な前提で扱い、実データが入ったDBに対して実行しない。

### 39.4 今回追加しないもの

- 本番運用の監視、バックアップ、障害対応設計
- デモDBリセット用の管理画面
- 認証なし公開
- 実データ入力を前提にした利用規約や同意画面

商品写真は `SUPABASE_STORAGE_BUCKET` 設定時にSupabase Storageへ保存する。未設定のローカル開発環境では、従来通りローカルファイル保存を使う。

## 40. フェーズ3-R バーコード出入庫

フェーズ3-Rでは、バーコードで商品を特定したうえで、スタッフが数量と理由を確認し、入庫または出庫を確定できる画面を追加する。

目的:

- 使用時や納入時に、商品名検索ではなくバーコード読み取りから在庫を増減できるようにする
- 読み取りだけでは在庫を変更せず、スタッフが確定ボタンを押したときだけ在庫を更新する
- 誤スキャン、モード間違い、数量入力間違いによる在庫ずれを避ける
- 入出庫の理由と読み取ったバーコードを履歴に残し、あとから確認できるようにする

### 40.1 画面

画面名は「バーコード出入庫」とする。

URL候補:

```text
/barcode/stock
```

既存の `/barcode` は商品確認、未登録バーコード整理、読取履歴保存の入口として残す。  
`/barcode/stock` は在庫数を増減できる専用画面として分ける。

### 40.2 基本フロー

基本フロー:

1. バーコード入力欄にカーソルを置く
2. バーコードスキャナーまたは手入力でバーコードを読み取る
3. 商品が1件に特定できた場合だけ、商品情報と在庫操作フォームを表示する
4. 現在庫、最低在庫、保管場所、読み取ったバーコードを確認する
5. `出庫` または `入庫` を選ぶ
6. 数量を入力する
7. 理由ボタンを選ぶ
8. 必要であれば補足メモ欄に補足を書く
9. `出庫を確定` または `入庫を確定` を押す
10. `StockItem.quantity` を更新し、`StockMovement` に履歴を残す

読み取っただけでは在庫を変更しない。必ず利用者の確定操作を必要とする。

### 40.3 表示する商品情報

商品が1件に特定できた場合、次を表示する。

- 商品名
- 商品コード
- JANコード
- 読み取ったバーコード
- 現在庫
- 最低在庫
- 保管場所
- 主発注先
- 在庫状態ラベル

GS1形式で有効期限、ロット番号、シリアル番号を解析できた場合は、既存の `/barcode` と同様に表示だけ行う。フェーズ3-Rでは、ロット別在庫や期限別在庫には反映しない。

### 40.4 入出庫操作

操作種別:

- `出庫`
- `入庫`

数量:

- 初期値は `1`
- 1以上の整数だけを許可する
- `+`、`-`、数値入力で変更できる形を検討する
- 出庫時は現在庫を超える数量を禁止する

確定:

- 出庫確定時は `StockMovement.movementType = OUT`
- 入庫確定時は `StockMovement.movementType = IN`
- `StockMovement.sourceType` は `BARCODE_STOCK` 相当の値を使う
- 変更前数量、変更後数量、操作者、日時、理由を履歴に残す
- 読み取ったバーコードは `StockMovement.memo` または `sourceId` 経由で追えるようにする

最初のMVPでは、`BarcodeScanLog` との厳密な紐づけは必須にしない。後続フェーズで、読取履歴を作成して `StockMovement.sourceId = BarcodeScanLog.id` とする方式を検討する。

### 40.5 理由ボタンと補足メモ

理由ボタンは必須とする。補足メモは任意とする。

出庫理由:

- 使用
- その他

入庫理由:

- 納品
- 返品戻り
- その他

補足メモ欄は、理由ボタンに補足を足すために使う。

例:

```text
理由ボタン: 使用
補足メモ: 午前診療分
履歴に残す理由: 使用: 午前診療分
```

補足メモが空の場合は、理由ボタンの値だけを履歴に残す。

補足メモ欄には、個人情報や秘密情報を書かない注意を表示する。

### 40.6 安全ルール

- バーコードを読み取っただけでは在庫を変更しない
- 商品が1件に特定できない場合は、入庫・出庫操作を表示しない
- 未登録バーコードでは在庫を更新しない
- 複数商品に一致した場合は在庫を更新しない
- 取込サンプル一致だけでは在庫を更新しない
- 出庫時は現在庫を超える数量を禁止する
- 数量0以下、小数、数値以外は不可とする
- 理由ボタン未選択では確定できない
- 確定中は二重送信を防ぐ
- 確定後は最新の在庫数を再表示する

同じ商品を短時間に連続して操作する場合の注意表示は、後続フェーズで検討する。フェーズ3-R MVPでは、まず1回ずつ確認して確定する運用にする。

### 40.7 今フェーズで作らないもの

- スキャン即時出庫
- スキャン即時入庫
- 連続入出庫モード
- 包装単位、箱、袋、本数の自動換算
- バーコードごとの既定入出庫数量
- 納品書との照合
- 発注候補や発注書との自動連動
- ロット別在庫
- 期限別在庫
- 使用期限アラート
- 外部発注送信
- メール送信
- 発注書PDF生成
- 納品確認

バーコード出入庫は、まず「商品を特定し、人が数量と理由を確認して在庫を動かす」範囲に限定する。

## 42. 管理者ユーザー管理

2026-05-21時点で、退職者対応と共用アカウント運用からの脱却を目的に、管理者向けのユーザー管理画面を追加する。

### 42.1 役割

`User.role` は次の2値を使う。

- `ADMIN`: 管理者。管理者向け画面にアクセスでき、同じ組織のユーザーを管理できる。
- `STAFF`: スタッフ。通常業務画面を利用できるが、管理者向け画面にはアクセスできない。

既存データで小文字の `staff` などが残っている場合でも、アプリ側では `ADMIN` 以外を `STAFF` として扱う。

### 42.2 有効・無効

`User.isActive` を追加し、ログイン可能なユーザーかどうかを管理する。

- `isActive = true`: ログイン可能
- `isActive = false`: ログイン不可

NextAuthのCredentials認証では、メールアドレスとパスワードが正しくても `isActive = false` のユーザーはログインを拒否する。

### 42.3 管理画面

管理者向け画面は `/admin/users` とする。

この画面では、同じ組織のユーザーに対して次を行える。

- ユーザー追加
- ユーザー無効化
- パスワードリセット

管理画面へのアクセスは、Next.js Proxyで `/admin/*` を保護し、さらにページとServer Action側でもログインユーザーの `isActive` と `role = ADMIN` を確認する。

### 42.4 ユーザー追加

管理者は、表示名、メールアドレス、初期パスワード、役割を入力してユーザーを追加できる。

- メールアドレスは重複不可
- パスワードは既存のパスワード仕様に従う
- パスワードはbcryptでハッシュ化して保存する
- 新規ユーザーは、同じ組織の有効なクリニックに割り当てる
- メール送信や招待メールはこのフェーズでは行わない

### 42.5 ユーザー無効化

管理者は、同じ組織のユーザーを無効化できる。

- 無効化は削除ではなく `isActive = false` の更新とする
- 自分自身のアカウントは無効化できない
- 別組織のユーザーは操作できない

### 42.6 パスワードリセット

管理者は、同じ組織の有効なユーザーに対して新しいパスワードを設定できる。

- メール送信は行わない
- 新しいパスワードは既存のパスワード仕様に従う
- 新しいパスワードはbcryptでハッシュ化して保存する
- 無効化済みユーザーのパスワードリセットは行わない

### 42.7 初期データ

ローカルseedの初期ユーザーは `role = ADMIN`、`isActive = true` とする。

既存の本番データでは、利用者が必要に応じてSupabase上の初期管理者ユーザーを `ADMIN` に更新する。

### 42.8 安全ルール

- パスワードの平文はDB、ログ、ドキュメントに保存しない
- 管理者画面はClient Componentだけの表示制御に依存しない
- 管理者Server Actionでは、必ずサーバー側でログインユーザーと組織スコープを確認する
- 患者情報、実在クリニック名、実在スタッフ個人情報はテストデータに入れない

## 48. 在庫直接編集の楽観ロック

2026-05-21時点で、在庫一覧 `/inventory` の数量直接編集に楽観ロックを追加する。

目的:

- 同じ在庫行を複数タブ、または複数スタッフが同時に開いた場合の後勝ち上書きを防ぐ

- 在庫変更履歴に、実際には反映していない競合更新を残さない

### 48.1 対象画面

対象は `/inventory` の在庫直接編集とする。

`/quick` の `+1 / -1` は数量の増減を原子的に行う既存仕様を維持し、今回の楽観ロック対象には含めない。棚卸セッションは別途セッション単位の入力・確定フローを持つため、今回の対象には含めない。

- `StockItem.quantity`
- `StockItem.updatedAt`

サーバー側では、更新前に現在DB上の値と比較する。

### 48.3 競合時の挙動

表示時点の値と現在DB上の値が一致しない場合は、在庫を更新しない。

競合時は、次の趣旨のエラーメッセージを表示する。

```text
他のスタッフが先に在庫を変更しました。最新の在庫を確認してから再度操作してください。
```

競合時は `StockMovement` を作成しない。

### 48.4 更新時の安全ルール

- Server Action側で必ず競合判定を行い、Client Componentの表示制御だけに依存しない
- 競合判定に失敗した場合は、在庫数も履歴も変更しない
- 既存の認証、組織、クリニックスコープ確認を維持する
- 秘密値、患者情報、実在医院名、実在スタッフ個人情報は扱わない
